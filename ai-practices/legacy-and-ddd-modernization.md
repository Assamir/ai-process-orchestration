# Legacy & DDD Modernization: Good vs. Bad AI-Engineering Practices

A reference for using AI agents to understand, model, and modernize legacy systems — distilling the domain before delegating implementation, and changing code in small, reversible, verifiable steps rather than sweeping rewrites.

## ✅ Good practices

- **Use LLMs as domain-modeling coaches to extract ubiquitous language cheaply** — A large model has absorbed hundreds of business-domain vocabularies, so it can simulate a domain-expert interview, translate and disambiguate terms, and check whether a name in code means what the business intends — collapsing the cost of workshops and experts to near zero.
- **Detect bounded-context seams where one term means several things** — Wherever the same identifier silently means different things (e.g. `Account` as user profile, bank account, and ledger account at once), a single hierarchy is conflating distinct concepts; splitting it into `UserProfile`, `BankAccount`, and `LedgerAccount` is a prime refactor, and AI helps find and name these seams.
- **Make the agent produce a plan/map, not production code, during diagnosis** — Separating diagnosis (map of domain, invariants, leaks) from implementation keeps the agent honest about what it actually verified and yields reusable markdown artifacts (e.g. `01-domain-distillation.md`) that feed later planning. Every diagnostic prompt should state "your product is a MAP/PLAN, not code — do not modify production code."
- **Score invariant candidates on multiple axes and pick the most-core-yet-least-enforced** — Rating each business rule on how core it is, how smeared across layers it is, and how weakly it is enforced surfaces the rule that hurts most — often not the one you'd first suspect.
- **Build invariants into a single guardian aggregate so rules cannot be bypassed** — In legacy code a business rule is "everywhere and nowhere," smeared across UI, services, and call ordering with the client as the only real enforcer; centralizing it in one aggregate root with precondition methods that throw named domain errors turns a remembered convention into code that cannot be circumvented.
- **Wrap leaky external dependencies in an Anti-Corruption Layer behind a narrow port** — When a library's types leak into the DB schema, API responses, and UI props, swapping it becomes a system-wide sprint and server-only code can bleed into the client bundle; an ACL value object plus a narrow port/adapter confines all knowledge of the dependency to one place.
- **Distinguish real technical debt from cheap, CI-caught coupling** — The most dangerous debt is quiet — two model halves nothing keeps in sync, an order-dependent DB write no compiler guards — while loud-looking, code-generated coupling caught by CI is actually safe; a report should separate the two so effort goes where fragility really is.
- **Use git history to distinguish accidental from essential complexity before changing code** — Code that looks like garbage is often a deliberate decision the system rests on; git archaeology (`log -L`, blame, commit/PR rationale) reveals intent the code alone hides. Positional column arrays that looked like textbook accidental complexity turned out to be a 2020 batch-import performance decision with zero ordering bugs in six years.
- **Right-size the refactor to the problem** — Match the scale of intervention to the problem: small modules stay simple, deliberate constraints get a guard, and rebuilds are reserved only for accidental complexity whose cost genuinely grows. Skip this filter and you get patterns everywhere, value nowhere.
- **Prefer incremental, reversible migration strategies over a big-bang rewrite** — Strangler Fig grows new code at the edges of the old, and Branch by Abstraction keeps the main branch releasable throughout; both keep a working, shippable whole at every moment, drastically reducing the risk a from-scratch rewrite carries.
- **Order a large change from leaves to root via try-revert-record (Mikado)** — Attempt the target change, observe what breaks, write it down, undo, then solve from the farthest leaf toward the root; this turns one big interdependent change into a graph of small, individually verifiable and reversible steps.
- **Compensate for stack friction via agent config rather than forcing migration** — For brownfield projects that fail some gates, documenting naming conventions, directory structure, middleware/error patterns, and enforcing JSDoc in `CLAUDE.md`/`AGENTS.md` closes the agent's knowledge gaps without rewriting the codebase — where most existing projects realistically land.
- **Adapt the planning workflow to brownfield by framing changes as a delta** — Existing-system work needs different questions (what hurts now, the smallest verifiable change, blast radius) and explicit capture of what must stay unchanged; brownfield mode reframes phases toward "what you change in what you already have" and adds "## Current System" and "## Constraints & Preserved Behavior" sections.

## ❌ Anti-patterns

- **Don't trust the folder structure as the data flow** — Directory layout can imply a thick stack of meaningful layers while a runtime trace reveals most are transparent pass-through wrappers and real work happens in one dense spot; the batch-write reality was hidden behind a single-save facade, knowable only by reconstructing the flow step by step.
- **Don't model the domain on day one when it is still shallow** — Early on, "hard" decisions about aggregates and boundaries are just guesses at a first impression of reality that almost always proves wrong, so heavy modeling invests in knowledge half of which expires before use; instead code, learn the domain, and model only when it starts to hurt (3x`Account`, a PRD entity absent from code, weekly migration-script rescues, a library swap ballooning into a sprint).
- **Don't confuse refactoring's question with domain modeling's question** — Refactoring asks "how do I safely change this code?" while DDD asks "does this code match how the business works?"; conflating them yields a credible-sounding plan that cements the chaos, because it designs around names that are themselves the mistake.
- **Never delegate a refactor blindly with "refactor this module"** — An open-ended refactor prompt produces an impressive but dangerous diff — e.g. 40 files with prettier names — that has no target shape, no historical justification, and no reversible path, so you can neither review nor roll it back piecemeal.
