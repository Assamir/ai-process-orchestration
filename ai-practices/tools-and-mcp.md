# Tools & MCP: Good vs. Bad Agent-Engineering Practices

Designing tools and MCP servers for agents is a discipline distinct from classic API design: the consumer has no docs, can pass anything, and reasons probabilistically. The practices below capture what makes agent tooling reliable, safe, and context-efficient — and the anti-patterns that quietly break it.

## ✅ Good practices

- **Don't map an existing API 1:1 onto agent tools** — APIs assume deterministic callers with documentation; an agent has neither, so design tools for the agent's needs, audit the underlying API first (missing actions, async/polling, rate limits, id-vs-name references, pagination), and build a proxy/MCP layer to reshape what you can't change.
- **Consolidate scattered API resources and granular actions into a few coherent, configurable tools** — merging related endpoints (statuses, labels, teams) into one configurable tool, or collapsing 13 filesystem actions into four, lets the model fetch exactly what it needs in one step and improves selection accuracy.
- **Optimize for the right balance, not raw tool count** — minimizing schema count is not the goal; aim for the best balance between available actions and how reliably the model uses them.
- **Cap an agent at ~10-15 tools and load schemas lazily** — tool schemas consume context and distract even when unused, so limit resident tools and use sub-agents, code-execution, or on-demand schema fetching (progressive disclosure / a ToolSearch-style lookup) to keep the base prompt lean.
- **But don't treat "10-15 tools" as dogma** — optimal count is highly context-dependent: a single CLI tool can unlock enormous capability while three poorly-chosen tools overwhelm an agent, so experiment rather than anchor on a rule of thumb.
- **Omit rare or destructive actions from the toolset** — excluding rarely-used or human-doable actions (delete_project, add_label) cuts context noise and raises tool-selection accuracy.
- **Write self-explanatory, namespaced tool names and high-signal descriptions** — a reader with no system knowledge should understand each tool; prefer `send_email` over `send`, prefix across servers (`resend__send`, `gmail__search`) to avoid collisions, and keep descriptions concise with only information that raises correct-selection odds.
- **Add a description and agent guidance to every schema field, using simple names and minimal payloads** — documented properties plus mode/detail switches are what let a model use a tool correctly with minimal extra prompting, while small payloads limit context consumption.
- **Decide per-field who fills it: model, code, or never the model** — design schemas by asking what the model must supply, what code should fill, and what the model must never touch, keeping security-critical values like permission-governing user IDs out of the model's hands.
- **Set sensible defaults and accept inputs in multiple forms** — since you can't force-correct an LLM's output, choose defaults carefully (assign to current user) and accept either an ID or a name so minor model mistakes are forgiven.
- **Auto-resolve and forgive model input mistakes inside the tool** — absorb realistic LLM error modes (resolving a path from just a filename, clamping out-of-range line requests, resolving message-vs-thread by ID in code) so hallucination-driven errors don't fail the task.
- **Return clear, actionable errors with recovery hints, not bare status codes** — agents lack API docs, so opaque errors doom tasks; say `team_id is invalid. Hint: fetch it via workspace_metadata`, `date range too large, max is X`, or even `did you mean...?` for likely typos.
- **Return dynamic, action-guiding hints in both success and error responses** — a hints/recoveryHints field ("File content was updated. Read it again.", available label values, applied corrections) surfaces environment context the model can't infer and reduces wasted steps.
- **Design tool outputs for agent self-correction and chaining** — on empty results suggest changing the query, explain auth/API errors with remedies, and on modify return exactly which fields changed so the agent immediately sees its effect and can chain the next action.
- **Use a shared, consistent response envelope across all tools** — a common structure (next_action, recovery, diagnostics) makes tool behavior predictable and gives the agent a uniform recovery surface.
- **Echo back useful state even when not strictly necessary** — returning the file path after a write (beyond a bare ack) reinforces the model's grounding so it can act on the modified resource next.
- **Let the agent control output verbosity/detail level** — letting the agent request how detailed results should be lets it manage its own context budget instead of always paying for maximum detail.
- **Treat validation and error handling as higher-standard than normal apps, and write descriptions for ambiguous natural-language input** — agents pass almost any data, often phrased loosely ("I need a 10-meter cable"), so validation, pagination, and descriptions must exceed everyday standards and anticipate free-form phrasing.
- **Build narrow, purpose-scoped tools instead of exposing raw API surface** — a pre-narrowed `gmail__search_support` keeps the agent inside its mandate, where a generic `gmail__search` invites scope creep and security risk.
- **Optimize the interface for smaller models as a forcing function** — tightening tools so weaker models succeed imposes useful constraints that also boost reliability with stronger models.
- **Guard destructive/lossy operations with checksums, dry-run, undo history, and read-only defaults** — high-risk write/edit/delete actions need rails that operate without the model: optimistic-locking checksums, dry-run preview, recoverable history, read-only DB access, and a reset/idempotency tool to bound blast radius.
- **Force JSON Schema with strict structured outputs instead of parsing free-form text** — strict mode guarantees the response shape, eliminating an entire class of ad-hoc parsing/validation bugs.
- **Generate MCP servers from a template via a spec-driven, AI-assisted procedure** — MCP servers are architecturally repetitive, so a reusable template plus a defined sequence (paste API docs, propose tools, group/prune, define I/O, implement, review, remove unused) makes building them fast and repeatable.
- **Treat MCP and native function-calling tools as one merged list** — tool origin is irrelevant to the agent, so merge native and MCP schemas/handlers into single lists; MCP is a complement, not a replacement.
- **Let tool descriptions and schemas be self-documenting** — names and JSON schemas already describe their usage, so repeating them in the system prompt wastes context and creates prompt/tool drift; only document genuine edge cases.
- **Choose CLI vs. MCP by fit-to-task, not hype** — CLI shines when you know the command and want auditable terminal history; MCP shines when the agent must discover what to run or runs shell-less. Use an MCP server when the agent needs structured, queryable data (Sentry issues/stack-traces) and a CLI when commands are known (sentry-cli releases); start with CLI and add MCP when a repeated discovery pattern emerges.
- **Use deterministic CLI tools to reduce noise outside the context window** — a command can traverse thousands of files, full git history, or the whole import graph outside the model's context and return only a condensed result, so the agent interprets distilled evidence instead of raw files.
- **Use cheap deterministic CLIs as a counterweight to the probabilistic agent** — split labor so git handles co-change, dependency analysis builds the graph, and ast-grep matches exact shape, giving reproducible answers on the axes the model is weakest at.
- **Pick the search family by question type rather than chasing one best tool** — lexical/pattern, structural/symbol, semantic/RAG, and agentic search each answer a different question ("where are endpoints?" lexical; "who imports payments?" dependency graph; "where is this type defined?" symbol search); match the family to the question instead of ranking tools.
- **Pair structural search with plain grep to catch false zeros** — a structural matcher returns zero both for truly no matches and for a wrong pattern, so confirm every zero with a cheap text grep before reading it as "no occurrences."
- **Know each diagnostic tool's blind spots and capability flags** — misreading capabilities wastes effort: wrangler tail omits HTTP status/body, and Playwright console/network reading needs no `--caps` flag while `--caps=network` mocks routes rather than reading them.
- **Split the workload: deterministic checks first, send only the ambiguous remainder to the LLM** — push rule-based verification (range checks) into cheap code and reserve the model for genuinely language-dependent judgments, rather than feeding 10,000 files to an LLM.
- **Generate-and-run code in a sandbox to process data at scale** — let the agent write aggregation scripts in a sandbox (e.g. Deno) so huge inputs never enter context and arithmetic stays deterministic, cutting cost, latency, and hallucination.
- **Offload bulk search of large data to a search tool, subagent, or cheaper model** — give the agent a search tool instead of holding raw logs in the main context; a dedicated subagent or cheaper model keeps the primary context clean and lowers cost.
- **Choose RAG architecture proportional to scale, combining lexical and semantic retrieval** — don't default to Elasticsearch/Qdrant/Neo4j; a filesystem agent or SQLite (FTS5, sqlite-vec) often suffices, and running keyword + semantic queries fused with Reciprocal Rank Fusion improves cross-language recall. Reserve graph-DB RAG for genuine multi-hop relationship needs.
- **Restrict agent database tools to read-only and add integrity-check tools** — limiting Cypher to read-only narrows blast radius, while dedicated audit/merge tools let the agent self-verify graph correctness (orphan nodes, duplicates).
- **Always fetch full message content before drawing conclusions** — when a search API returns metadata-only listings first, guessing answers from subject lines instead of retrieving the body causes wrong extractions in agentic loops.
- **Implement inter-agent communication as explicit delegate and message tools** — a `delegate` tool that opens a sub-thread (own system prompt and toolset) plus a `message` tool for two-way contact gives a clean, debuggable mechanism where the sub-agent's reply becomes the caller's tool result.
- **Provide a dedicated `think` tool to create reasoning space** — a no-op tool whose only purpose is to let the model pause and self-question measurably changes downstream behavior and helps it spot gaps between what it knows and what it could know.
- **Pin tool/library versions the model knows best, and tell the agent what's available** — choose well-represented versions the LLM handles reliably (Tailwind v3 over v4) rather than the newest, and explicitly declare the curated toolset/libraries, since undeclared capability is wasted.
- **List tools in AGENTS.md — it measurably increases their use** — the rules file genuinely steers behavior; one study measured a 1.6-2.5x increase in usage of tools mentioned in AGENTS.md, making a curated list a real lever.
- **Choose the model and storage format that fit the problem, not convenience** — multimodal capability, speed, price, privacy, and local-run feasibility vary widely (check leaderboards like Artificial Analysis); likewise markdown is ideal for AI but Notion/Docs win for concurrent multi-user editing, and conversion loses too much to substitute.
- **Give agents a vision/analysis tool to inspect images they produce** — generated images are tool outputs not auto-fed to the model, so an image-analysis tool closes the perception loop and enables critique-and-retry.
- **Generate PDFs from your own templates rather than parsing arbitrary ones** — producing PDFs (HTML-to-PDF via Puppeteer over a known template) is far more reliable than later extracting structured content from arbitrary PDFs, a genuinely hard problem.
- **Embed images as remote links, not local files** — linked remote images can be referenced and cited by the agent in its outputs, whereas locally saved files the agent cannot reference are limiting.
- **Auto-invalidate trusted-tool exceptions when a tool's signature changes** — allow-listing a tool to skip approval is convenient, but if its name, description, or schema changes (common with silently-updating MCP servers) the prior trust must be revoked automatically, and tools should be allow-listed by a collision-free composite key (server + action, e.g. `resend__send`).
- **Constrain an agent to a tool family via prefill/constrained decoding** — when you can deterministically detect the start of a related action chain (a browser session), prefilling the model's tool-call reply with `browser_`-prefixed tokens restricts it to relevant tools without deleting other context.
- **Treat external context and filesystem access as an untrusted attack surface** — tool-supplied content (emails, docs, web pages) can carry prompt-injection instructions the model may follow, analogous to XSS from unvalidated input; limit and verify external sources, and scope filesystem permissions tightly since an agent might even accidentally delete a directory.
- **Prefer integrating into tools the client already uses, but know your host's protocol gaps** — extending existing ecosystems (Linear, Slack, Claude) lowers adoption friction and leverages existing contracts; first verify the host actually implements the MCP features you need (sampling, system-instruction control), since most clients don't fully implement the protocol.
- **Inventory your real tools and inspect their APIs for AI-fitness before deciding where AI helps** — catalogue the tools you actually use and their integration surfaces (APIs, webhooks, deep-links), and verify endpoint structure and returned payloads, since an API's mere existence doesn't mean its data shape is complete enough for an agent. Models are poor advisors here — their suggestions are too generic.
- **Prefer a purpose-built tool/MCP over free interpretation of docs, designed around business processes** — when you control neither the doc interpretation nor the user query, an internal MCP fitted to the process is justified; design dedicated interfaces around processes (sales monitoring, task management) and expose deterministic, code-backed actions via generative UI (MCP Apps) rather than free-text calls for sensitive actions.
- **Package recurring integrations as CLI tools or MCP servers for portability** — wrapping everyday services into a CLI or MCP makes capabilities easy to move between agents and share across a team, and adding tools alone can transform what an agent does without changing its core logic.
- **Design for provider portability from the start** — locking into one provider blocks adopting much better models elsewhere; build a translation/mapping layer (AI SDK, LiteLLM, or your own format) so switching is cheap, accounting for cross-provider differences in system-message handling, reasoning settings, and thought signatures.
- **Be skeptical of hyped, immature AI tooling** — many loudly-promoted AI tools are abandoned early and unstable frameworks can lock you out of newest provider features, so pick deliberately and keep a healthy distance even from tools you haven't run in production.
- **Use established libraries for streaming-Markdown rendering** — streaming output creates genuinely new UI problems (incomplete Markdown, per-token re-render) that purpose-built libraries (remend, marked alongside markdown-it/highlight.js) solve far more reliably than hand-rolled code.
- **Restrict STDIO MCP servers to user-device desktop apps only** — letting users attach local STDIO MCP servers is safe only when the process runs on the user's own machine; doing so in a hosted production app is a security risk.
- **Give agents tooled, scoped access to the issue tracker for low-risk automation** — connecting an agent to the backlog via MCP/CLI to triage, link PRs, and report is one of the most accessible AI automations, since even mistakes are cheap to revert and never break production code.
- **Use AI-native search and inject live library docs instead of raw HTML or model memory** — agentic search returns clean markdown with citations and answer endpoints, and tools like Context7 pull current, token-budgeted library docs into context so the agent uses real APIs (`createEmptyCard`, `fsrs().next()`) instead of hallucinating functions; prefer agent-friendly markdown (llms.txt, `.md` URL suffixes) over noisy HTML.
- **Run setup/config commands outside the agent session** — MCP and tooling setup that changes the agent's own configuration should run outside the running agent so a fresh session picks up the updated tool set cleanly.
- **Verify a tool's context-injection support before relying on agent self-correction** — some tools can block an action but can't tell the agent what failed (Windsurf hooks exit 2 with no context), making automatic correction impossible; check this before designing a self-healing loop.
- **Repurpose browser-automation tooling for evidence collection** — the same Playwright tooling used for E2E tests can feed the agent console messages and network traffic during a live repro, cleanly separating client-side from server-side faults.
- **Give the agent a semantic accessibility-tree view instead of pixels, and prefer CLI over MCP for browser automation** — a compact accessibility-tree snapshot (roles, names, refs like `e15`) lets the agent navigate without a vision model and steers it toward role-based selectors; for a coding agent, a Playwright CLI writing YAML snapshots to disk used ~27K tokens per scenario vs ~114K for MCP.
- **Encode the method's discipline in tool/phase constraints to prevent shortcuts** — limiting the available toolset to roles valid for the current phase makes the method itself enforce correct ordering, so neither human nor agent can skip steps.
- **Audit skills before installing them as you would any software** — skills can contain executable code in `scripts/`, so read SKILL.md, inspect what scripts run and touch, and verify the source; treat installation as a supply-chain risk and never install what you haven't read.
- **Treat the allowed-tools allowlist as a hard security boundary** — declaring the permitted tools in a skill makes the permission surface an explicit, auditable contract that bounds the agent's blast radius; changing it is a chain-breaking change, not a convenience.
- **Write host-agnostic skills with tool self-discovery and graceful fallback** — rather than hard-coding one harness's tool name, describe the tool's shape, scan available tools by capability pattern (ask/question/input, subagent), and degrade to plain chat if none exists, so the same procedure ports across Claude Code, Cursor, and Codex.
- **Provide platform-specific command blocks instead of relying on shell translation** — auto-translating shell commands across platforms can silently corrupt them (a bash `while read` loop becoming a literal `B:$f` that Windows reads as drive B:), so supply verified per-platform (bash / PowerShell) blocks.
- **Rely on auto-generated contracts for cross-repo visibility, not a hand-built meta-context** — a central meta-context describing many independently-changing services rots fast; CI/CD-generated API contracts (OpenAPI, GraphQL SDL, Protobuf), test specs, and code docs convey how a module works more cheaply and reliably. Note also that tool instruction files merge additively (root-to-cwd concatenation, not nearest-wins).

## ❌ Anti-patterns

- **Don't blindly adopt official MCP servers or CLIs when you need custom behavior** — ready-made integrations optimize for generality, not your app's constraints, so wiring an agent to an official server (e.g. Replicate) is the wrong default when you need personalized, strictly-fixed behavior.
- **Don't accept the first LLM-proposed tool schema — it's only an illusion of correctness** — initial AI suggestions look plausible but silently omit critical fields (pagination, multiple recipients, attachments, read state, real date fields), yielding tools agents can't use well.
- **Never return large binary blobs like base64 attachments in tool output** — returning an attachment as base64 floods and kills the model's context; return a link or reference to the resource instead.
- **Don't grant agents full broad access to sensitive accounts** — full mailbox access is a poor idea; scope tools to read only selected labels or create drafts, since least-privilege design contains both error and security risk while preserving usefulness.
- **Don't connect tools without a concrete use case and personalized procedures** — merely wiring up an integration (a calendar MCP just to add an entry) can be less effective than doing the task by hand; value comes only when usage is personalized via process descriptions, procedures, and scripts.
- **Don't host an agent inside MCP when you need control of the user-interaction surface** — behind MCP tools in a third-party client you can't reliably request confirmations, report background-task status, or implement rich permission rules, breaking any human-in-the-loop flow.
- **Don't overload one MCP server with many services** — connecting a single server to more than one service is acceptable but rarely seen, sharply increasing complexity and reducing flexibility, so do it sparingly and deliberately.
- **Don't mislabel structural search as semantic/RAG to the agent** — tools like Probe advertise "semantic code search" but return complete functions/classes via AST parsing, not embeddings; calling a precise structural result a probabilistic semantic match makes the agent treat exact matches as approximate similarity, giving its map the wrong kind of confidence.
- **Don't paste a question into a free chatbot and trust the first answer** — chatbots answer from training memory, hitting the data cutoff and producing hallucinated, stale package names; their answers are ungrounded and unverifiable. Note that embedding models share this knowledge-cutoff limitation, so semantic search silently fails on terms outside its training data.
