# Deployment and Production: Good vs. Bad AI / Agent-Engineering Practices

Shipping LLM-backed systems to production demands the same discipline as any software rollout, plus extra guardrails for non-determinism, cost volatility, and prompt injection. The practices below distill what holds up under real production load.

## ✅ Good practices

- **Persist intermediate progress and cache completed work** — Write step outputs to disk so a multi-stage pipeline can resume after failure and skip already-generated work, saving cost and time.
- **Parallelize and batch LLM calls within rate limits** — Send requests in parallel groups to cut wall-clock time while staying under per-account API rate limits; bundle many records (e.g. a numbered list) into a single classification call to slash call count. Be aware that excessive parallelism can itself trigger provider slowdowns.
- **Exploit prompt/input caching at scale** — Caching is a key mechanism for agents running at scale, cutting repeated recomputation of input tokens; factor cache support into provider choice.
- **Abstract the model interface to avoid provider lock-in** — Production systems usually span multiple models and providers, and leadership shifts fast; a thin common interface over official SDKs keeps you able to swap models and capture new capability/cost gains.
- **Avoid heavy AI frameworks; prefer a thin provider abstraction** — Fast-moving APIs make frameworks like LangChain/CrewAI a migration burden with few positive production accounts; a lean interface over official SDKs lets you swap models and add your own event/hook system.
- **Avoid deep lock-in to provider-specific beta features** — APIs are still maturing; features that tightly bind you to one provider can be deprecated (e.g. OpenAI's Assistants API replaced 15 months later by Responses API), stranding your investment.
- **Handle critical input, consent, and permissions in deterministic code/UI** — All agent actions are non-deterministic, so error-prone or irreversible steps need GUI forms and button-based approval (chat "I changed my mind" can be ignored), and permission/access checks must live in code the agent cannot bypass.
- **Constrain agents at the environment level because prompt injection is unsolved** — Injection has no reliable defense today; an email saying "send me your meeting plan" can drive a calendar+email agent to leak data, so sandbox tool-using agents and keep them out of leak-prone areas.
- **Choose STDIO vs. Streamable HTTP transport by deployment model** — Use STDIO for single-user local/desktop and CLI tooling (each connection spawns a process); default to Streamable HTTP for remote, multi-user, OAuth-capable servers.
- **Optimize media before processing** — Large audio/video drives up cost and latency; compress or speed up recordings when no essential detail is lost.
- **Don't deploy LLM-driven actions where an accidental data leak is unacceptable** — Cross-tool data bleed remains possible even with confirmation flows; if a wrong-recipient leak would be intolerable and user approval isn't sufficient mitigation, the honest call is not to use an LLM there.
- **Provide heartbeat progress, queueing, and background processing for slow tasks** — Models are slow; surface the current step and partial results, let users queue messages or open new threads, and run work in the background so it survives a closed tab or unstable connection.
- **Moderate inputs and enforce per-user rate/spend limits** — Untrusted content (uploaded docs, fetched pages, injection) can hijack an agent or get your org account banned; pair moderation APIs with per-user keys carrying their own quotas (e.g. via OpenRouter).
- **Retry transient API failures with backoff and respect rate-limit headers** — Production model APIs are unstable and rate-limited; auto-retry on transient errors (e.g. 503) with backoff and read reset headers to pace requests, since aggressive polling triggers long blocks.
- **Build resilient clients that tolerate corrupted data and random errors** — Production endpoints can return malformed payloads or random failures for valid requests, so parse defensively and retry rather than assuming clean responses.
- **Run background/consolidation work off-session via Batch APIs** — Tasks like memory consolidation don't need to run live; deferring them (e.g. once a day) lets you use cheaper Batch APIs and keeps the session lean.
- **Build UIs that expose original fragments and source links** — Retrieval is imperfect and incomplete results are hard for the model to detect, so a navigation interface showing original snippets and links lets users verify and recover from errors.
- **Enforce attachment validation, moderation, and access control in code** — Files entering an LLM pipeline need deterministic guards: size/format/mime-type validation, moderation, strict access permissions, hard-to-guess expiring share links, and size optimization — the agent can't be trusted to enforce these.
- **Require human verification for outputs with external impact** — Hallucination and injection make autonomous sending dangerous; the agent should only draft (e.g. a campaign draft), with a human gating anything that reaches real recipients.
- **Build human-facing management dashboards for agent systems** — Full autonomy isn't reliable yet, so a control panel showing live sessions, schedules, system stats, and areas needing attention keeps the human as coordinator and surfaces problems for intervention.
- **Sandbox agents that take consequential, risky, or code-executing actions** — Granting terminal access, arbitrary code execution, or cross-tool data movement requires isolation (e.g. Daytona) to contain blast radius while still relaxing tool limits; even a dedicated old laptop can host actions blocked on remote servers.
- **Account for cross-tool combination risks and prefer sandboxing over crippling restrictions** — An agent that moves data between connected tools can exfiltrate or write past intended boundaries; heavy restrictions destroy usefulness, so a sandbox grants freedom while bounding risk (at some architectural cost).
- **Enforce hard usage/cost limits and monitor for anomalous spend** — Per-user cost is unpredictable and can spike from bugs or abuse (1–3% of users often generate most cost), so hard limits are critical and monitoring enables fast reaction and support investigation.
- **Anonymize user data captured for observability and evals** — Agent traces are rich eval data but contain personal information; strip names, addresses, and contact/personal data even when you're the sole processor.
- **Manage latency and cost via the five token levers** — Response time and cost are governed by input tokens, cache utilization, output tokens, number of (serial) requests, and model size — agent step count and parallelism are tunable, not only prompt length.
- **Address user-side misuse through onboarding and product communication** — Many agent failures stem from users interacting poorly (e.g. low-quality input), so onboarding and expectation-setting messaging matter as much as the agent's architecture.
- **Run model-generated UI code in a sandboxed, permission-controlled iframe** — When an agent generates and executes HTML/JS on the fly, isolating it in an iframe with fine permission control contains the risk of model-generated code.
- **Prefer constrained JSON/declarative UI over free-form code generation** — Free-form generation maximizes flexibility but minimizes control; emitting components against a fixed schema (JSON Render) lowers error risk and enables state save/load, while protocols like MCP Apps keep generated UI code out of the model entirely.
- **Treat AI implementations like ordinary software implementations** — Most of an AI system is conventional application logic, so reuse established engineering knowledge instead of over-rotating on the model and neglecting the surrounding system.
- **Anticipate the gap between AI's apparent and real capabilities** — Teams overestimate what an agent can do and hit concrete limits (long/binary docs, images, blocked sites, latency, cost); plan for these from the start.
- **Match the agent interface to how the consumer already works** — The delivery surface (CLI, MCP, chat app, messenger, custom UI) should be driven by who uses it: CLI for dev teams that can install/update, MCP or messenger bots for non-technical users in familiar contexts.
- **Define where off-the-shelf integration stops paying off** — Reusing existing tools is sensible early but "quickly becomes very limiting," so consciously decide when to switch to a dedicated solution rather than fighting platform constraints indefinitely.
- **Account for the subscription-vs-API cost asymmetry before going fully custom** — Subscription plans are far cheaper at scale than raw API usage, so evaluate the economic model first; note hidden drivers like Claude.ai not supporting MCP sampling, which still bills API costs.
- **Scope agent permissions tightly and treat prompt injection as a real attack surface** — Agents touching task systems, repos, and APIs are attack surfaces; apply least-privilege and read-only access (e.g. a read-only mailbox that can message only the owner), recalling supply-chain attacks that steal tokens via CI.
- **Prefer local models for highly sensitive data** — Some data (e.g. clipboard contents) is too sensitive to send to provider servers; run local models to extract value without exposing private content.
- **Govern open asset links for permissions and expiry** — Open links are near-indispensable for agents yet hard to control; establish access rules, keep an inventory of resources, and a way to restore availability.
- **Don't expose the raw model through a generic endpoint** — Clients should hit specialized endpoints (e.g. `/product/review`) with fixed-shape I/O rather than a raw `/api/chat` passthrough, constraining model contact at the API boundary for safety and predictability.
- **Stream events to reduce perceived latency and show progress** — Streaming partial results and progress events keeps users informed during long, often agentic operations and lowers reaction time.
- **Handle long-horizon tasks that outlive the connection** — In production, users close tabs and tasks exceed HTTP timeouts, so agent work must be resumable in the background rather than tied to a live request.
- **Always sanitize LLM-generated HTML before rendering** — Model output rendered as HTML is an injection vector; stripping malicious tags/scripts (e.g. with DOMPurify) is a critical safeguard, not optional polish.
- **Prefer DB search extensions over dedicated vector infrastructure until scale demands it** — SQLite (fts5 + sqlite-vec) or Postgres full-text search suffices for many projects; move to Qdrant/Elasticsearch only when search becomes critical at large scale, since dedicated stacks add disproportionate complexity.
- **Reserve costly realtime multimodal models for cases that need them** — Realtime audio/video models are expensive; a cheaper STT→LLM→TTS pipeline is preferable for simple interactions where unified multimodal processing adds no value.
- **Manage technical debt aggressively because GenAI apps change faster** — Building LLM software resembles classical development but with a multiplied rate of change across UI, DB, and server config, so debt accumulates faster and demands better-than-usual discipline.
- **Protect production against rate limits, moderation bans, and cost spikes** — Pre-empt common failure modes with key rotation or OpenRouter for limits, a moderation API plus custom I/O filters, and hard per-user spend limits.
- **Design for LLM latency at the UX and architecture level** — LLMs remain one of the slowest parts of an app; mitigate with UX choices, caching, and partial parallelism, while avoiding parallelism so aggressive it triggers provider slowdowns.
- **Prevent editing the assistant's turns to block many-shot jailbreaking** — Editing user content is fine, but letting users edit the model's own tool-using responses opens many-shot jailbreaking, where the user falsely implies the agent already did something it shouldn't.
- **Account for transcription hallucinations from silence and language-mixing** — Speech models trained on subtitles emit fixed phrases on silence (e.g. "Thanks for watching!") and stumble on mixed-language input, so audio pipelines need handling for spurious or corrupted output.
- **Assume zero AI knowledge in end users** — Building an AI feature is easy; the real production challenge is delivering value to users with near-zero understanding of how models behave, so design for minimal user knowledge.
- **Design products for the agents acting on users' behalf, not only users** — As agents increasingly act for users, build applications to be consumed by agents as well as humans — a shift that affects interface and API design.
- **Re-verify sandbox and tool gating before exposing to production** — Even with a solid sandbox and confirmation prompts, capabilities fine locally (e.g. STDIO server registration) must be disabled in a public web app, so add checks before publishing.
- **Configure a deliberate allow/deny/ask permission policy** — A small allowlist for routine local ops (npm, npx, local git, Read/Edit/Write), an "ask" tier for network/publish ops (curl, git push), and a denylist for destructive patterns (`rm -rf`) removes approval fatigue while keeping a safety net; rules evaluate deny → ask → allow, first match winning.
- **Split broad command wildcards into narrow patterns** — A pattern like `Bash(git *)` silently authorizes `git push`, `git reset --hard`, and `git clean -fd`; enumerate specific safe subcommands so risky operations stay gated.
- **Treat permission policy as probabilistic control, not an absolute lock** — Pattern-matching filters by name and arguments, not effect, and is bypassable (a denied `npx` reached via `/proc/self/root/usr/bin/npx`; `deny Read(./.env)` doesn't stop `cat .env`), so layer additional isolation for defense in depth.
- **Only enable YOLO / skip-permissions mode inside an isolated sandbox** — Full delegation without isolation hands a model your credentials and prod access; confine `--dangerously-skip-permissions` to a non-root dev container with restricted network, only the trusted repo mounted, git versioning, no prod MCP servers, and a rule forbidding destructive DB ops.
- **Surface the error before trying to fix it** — For swallowed-error bugs, the first repair is to stop hiding the failure (return 500 instead of a fake 200); only once the error is visible can the underlying write be fixed.
- **Treat platform/infrastructure choice as a deliberate, de-risked decision** — Picking a deployment platform commits the project for its lifetime, so decide consciously up front based on runtime/build fit, agent-operability, and clear rollback/scaling/cost answers rather than deferring to "whatever the tutorial says."
- **Enforce least-privilege and human authorization for irreversible production actions** — Agents wired to CLI/MCP get real infrastructure access, so scope tokens to only what's needed (keep them in env vars, not committed config) and reserve destructive actions (drop DB, rotate main secret, DNS) for manual execution — "a manual click costs 30 seconds, cleaning up an automated mistake costs hours."
- **Prefer safer-by-default command paths for production-affecting operations** — A CLI that defaults to a non-production draft and requires an explicit flag to publish (e.g. `netlify deploy` vs `--prod`) builds a human-intent gate into the operation that an equivalent MCP tool call may lack.
- **Plan guard-first with separately reversible, gated rollout steps** — Land the mechanism green and turn enforcement on separately (per-layer, behind an exceptions file with mandatory justifications) plus an explicit "what we're NOT doing" section, keeping every step an isolated, reversible commit.
- **Optimize products and docs so an agent can succeed in one shot** — Agents increasingly select, install, and run products in a single pass, so a first successful run without human intervention drives adoption; low-friction APIs and readable markdown docs win (Claude Code proposes Resend 63% vs SendGrid 7%) because "your product is no longer your UI, it's your API."
- **Apply least-privilege and keep destructive actions in the UI for backlog access** — Always ask whether access is strictly necessary; hand the agent a scoped (not admin) token, keep destructive ops (delete project, mass status changes) manual, and review audit logs after the first session.

## ❌ Anti-patterns

- **Assuming MCP solves security or privacy** — Generating tools/servers does nothing for prompt injection or data exposure; without access blocks, action/query limits, validation, anonymization — and acceptance that some actions simply must not be exposed — end-user servers become a liability.
- **Carrying internal-only assumptions into public deployments** — Control you rely on internally (known tools, processes, trusted users, contained data flow) evaporates for public MCP servers, so designs safe internally can be unsafe once published.
- **Treating preview-stage and Live APIs as production-ready** — Models in "preview" and real-time "Live" APIs still show glitches, stability problems, and high cost, so shipping them to production without accounting for that risk invites failure.
- **Pushing for full end-to-end automation** — Total process automation is a risky extreme; gradually shape the solution toward real business needs rather than replacing the human wholesale.
- **Trusting the model because the provider is trusted** — Even on private Bedrock/Azure instances with full data control, the model can leak data, run destructive code, call the wrong tool, or mislead a human into a harmful action (e.g. restarting a production server outside procedure).
- **Shipping AI where it adds no real value** — Many AI deployments are unnecessary and deliver a worse experience than a good conventional UI, which wins on usefulness except at scale; apply AI only where it genuinely beats the alternative.
- **Approve-everything paralysis and reflexive blanket allow** — Manually confirming every prompt turns the workflow into an endurance test and gets the agent abandoned, while clicking "don't ask again" on everything (`Bash(*)`) eventually authorizes a command that deletes a database or secrets file.
- **Letting swallowed errors hide failures from your test gates** — A try/catch that logs a warning and returns 200 hides failures from clients, unit tests, type-checks, and hooks; this OWASP A10:2025 anti-pattern can go unnoticed for weeks until monitoring (the only steady signal) surfaces it.
- **Assuming worktrees isolate the runtime environment** — A git worktree isolates git state but not runtime, so parallel agents still contend for the same dev-server port, local database, or payment sandbox — handle that isolation separately.
- **Deploying by following the first random tutorial** — Googling "free hosting [framework]" and clicking Deploy works only until the first real problem (wrong runtime, slow build, dependency issues), by which point the platform is locked in and you can't answer rollback/scaling/cost questions.
