# Planning, PRD, and Roadmap: Good vs. Bad AI-Engineering Practices

A reference for how to shape product intent, specs, and roadmaps when agents do the downstream work. The recurring lesson: invest in cheap, structured, traceable artifacts up front so every later agent step inherits a sound contract instead of confident guesses.

## ✅ Good practices

- **Interview before generating the spec, not after** — Run a Socratic Q&A that drills into gaps first; a model asked to write a PRD up front just fills holes with plausible assumptions the user never made.
- **Make open questions explicit instead of inventing answers** — Route every unresolved gap verbatim into an `## Open Questions` section rather than letting the model silently fabricate domain decisions.
- **Treat the PRD as a durable contract for downstream steps** — A stable written contract (user, problem, scope, non-goals, success criteria) anchors later prompts so the product doesn't drift into a collection of accidental features.
- **Adversarially self-challenge every requirement** — For each requirement, ask "what would have to be true for this to be wrong?" with domain-specific counter-arguments before locking scope, surfacing weak or unnecessary items early.
- **Detect "empty CRUD" and force a real domain rule** — When "business logic" is just add/list records, call it out by name and demand a concrete rule (generation, an accept/reject gate, an algorithm) that turns data into value.
- **Name known anti-patterns during the interview and offer concrete remediation** — Surface failure shapes like empty-CRUD or oversized-MVP by name, with specific scope-down moves, turning vague design risk into actionable feedback.
- **Surface the cost of an oversized scope but let the user choose deliberately** — Detect flows too big for an MVP, print the trap, offer scope-down vs. an acknowledged longer timeline, and record a dated acceptance line rather than nagging.
- **Defer downstream concerns out of the current artifact's scope** — Keep framework/database/deployment/file-path choices out of the PRD; park volunteered stack content in a forward block so a later specialized step owns those decisions.
- **Sequence discovery before document generation as an explicit skill chain** — Force idea → structured shape-notes → PRD before any code, so each stage's audited output is the next stage's input rather than a document built on unexamined assumptions.
- **Re-run lightweight shaping for every major change, not just at project start** — Do a quick mini shaping session per new feature or integration; treating shaping as a one-time ritual leaves later modules with no contract the agent can ground on.
- **Refuse to generate from low-quality input; score readiness first** — Score the PRD 0–4 on concrete signals and halt below threshold; a roadmap built from a hollow PRD inherits the holes as blocked slices and isn't a real execution plan.
- **Plan a sequence of vertical milestones before coding** — Act as a technical PM defining goal, sequence, risk, and ownership so agent delegation becomes a predictable contract ("milestone 1 of 5, here are its dependencies") instead of unverifiable progress.
- **Slice work vertically (end-to-end), not horizontally by layer** — Decompose into user-visible full-stack capabilities so agents get units they can implement and verify end-to-end; horizontal "API layer / DB layer" work is the anti-pattern to prevent.
- **Allow only foundation tasks that unlock a specific vertical slice** — Permit groundwork (auth, persistence, spikes) only when each foundation names the single downstream slice it unblocks, preventing horizontal drift disguised as a foundation.
- **Never let the agent invent scope beyond the source** — Every roadmap item must trace to a PRD user story or functional requirement; ideas surfaced mid-process become open questions or parked items, not silently added work.
- **Surface blocking unknowns and auto-block dependent work** — Mark any unknown with "Block: yes" to flip a slice to blocked, telling you what you don't know before implementation; the roadmap's value is partly in showing what is NOT yet plannable.
- **Keep project state in a shared backlog, not in heads and scattered DMs** — An external backlog with status, ownership, dependencies, acceptance criteria, and an audit trail is the operational memory of the project and reduces bottlenecks and "how's it going?" pings.
- **Plan as a control mechanism before the agent edits code** — Write the end state, phases, contracts, and success criteria first; this shifts the human review point earlier ("are these files in scope?") rather than reacting to an unreviewable diff afterward.
- **Match planning rigor to risk and file count, not a blanket rule** — Planning has overhead, so scale effort to the change: more risk and more files touched means a readable up-front plan pays off; low-complexity tasks get a quick plan and go.
- **Resolve propagating domain decisions before coding** — Some features are blocked by un-made domain decisions (e.g. the data-model shape) rather than missing code; settle these first so the agent doesn't pick a model-of-the-world that later collides with a library API.
- **Make a decision gate an explicit planning question** — When the contract declares a ranking or option a proposal, accepting it becomes a deliberate "which option do we implement?" question at planning time rather than a silent assumption.
- **Feed research into planning so plan questions become solution-oriented** — Built on gathered research, the plan states concrete decisions that each point to a source (the exact library, the rating scale), turning the plan-brief into a real decision contract.
- **Feed upstream diagnostic artifacts into downstream planning skills** — Pass a ready research/frame document as a parameter to a planning skill; it narrows and sharpens the output and reduces the clarifying questions the agent must ask.
- **Let research redefine the task rather than rushing to write a test** — A research step can split one risk into an already-protected invariant and a genuinely untested behavior, changing what the phase should deliver and avoiding testing the wrong half.
- **Gather evidence before deciding on any refactor** — Stop deliberately at analysis (a verified fragility map, no target architecture yet) so the later change decision rests on evidence instead of intuition.
- **Prefer structured, machine-readable artifacts over ambiguous ad-hoc answers** — An ad-hoc prompt yields prose with no file and no validation that automation can't consume; an output file with explicit frontmatter becomes a reliable contract for the next deterministic step.
- **Persist infrastructure decisions as a durable in-repo contract (ADR)** — Record platform, stack fit, secrets locations, rollback steps, agent permissions, and known risks so humans and the agent can revise them consciously later instead of rewriting in a panic.
- **Capture explicit out-of-scope exclusions with their rationale** — Record what should NOT receive budget and why, turning a forgotten decision into a documented one that answers a future "should we add X?" months later.
- **Refresh strategy artifacts through the same audited cycle as the original** — Route a `--refresh` through the same research-plan-implement loop so changes get an auditable trail of what changed and when, rather than ad-hoc edits.
- **Validate the premise with prototypes/MVPs before committing to a full build** — Map concrete use cases and test them cheaply so you can abandon or rescope an infeasible idea before investing in the full project.
- **Define an explicit minimal foundation as the starting point** — Anchor on a deliberately minimal, low-utility baseline to build from and iterate on, rather than chasing an unbounded ideal up front.
- **Start with one area or slice instead of building the whole knowledge base** — Pick a single appealing slice and ship it; a full taxonomy up front is nearly impossible, and the best ideas are a result of the work, not its precondition.
- **Map the process before applying AI, including where NOT to use it** — Answer "what do we NOT want to do?" and "how?" as deliberately as "what do we want to do?", explicitly including the choice to skip AI where its benefit is small or absent.
- **Validate assumptions against messy reality before automating** — Map the full process before handing it to agents; a poorly designed automated workflow whose assumptions fail on edge cases creates more problems than value.
- **Build support tools for existing workflows instead of full automation** — Augmenting human processes (labeling, drafting, enrichment) delivers bounded-risk value now, whereas end-to-end automation of relationship-building invites reputational and business harm.
- **Demo small prototypes to surface unforeseen use cases** — Showing even a tiny working tool to stakeholders reveals applications nobody imagined and generates demand for more, so small-scale experiments pay off.
- **Make the business aware of technical possibilities and limits** — Sound adoption decisions (vendor, cost, legal) depend on the business understanding what the technology can and cannot do, including that 100% reliability is currently unlikely.
- **Build for the case where models keep getting better** — Design so future model improvements strengthen the product; avoid building in areas where current LLMs are only mediocre, since the next release can make that effort worthless.
- **Right-size eval investment to project maturity** — Treat eval-writing as a core skill, but for fast-changing early-stage products it can be premature; match the investment to stability, cost priorities, and stage, like the choice to invest in E2E tests.
- **Start minimal and add quality layers as gaps appear** — Treat quality layers as a menu, not a mandate; begin where cost-to-signal is best (one lint hook, one commit gate) and add layers based on observed escaping errors.
- **Parallelize only independent slices, after an explicit check** — Run agents in parallel only on tasks that can reach PR without colliding on shared files, contracts, or layers; ask the agent to assess this up front to choose parallel vs. sequential.
- **Capture incidental improvements as new roadmap slices, not scope creep** — Record problems noticed mid-implementation as a distinct, parallelizable roadmap entry rather than bolting them onto the in-flight slice, keeping each unit focused and conflict-free.

## ❌ Anti-patterns

- **Faster is not better when the direction is wrong** — Skipping planning to jump into code only accelerates delivery of bad decisions; an empty initial contract makes every downstream automation faster only at producing the wrong thing.
- **Ritualized over-process is as harmful as plan-less vibe coding** — Both "just build everything from the roadmap" (large unreviewable diffs) and a fixed heavy ceremony of dozens of prep steps fail; the latter "sounds mature but turns the workflow into art for art's sake."
- **Don't have agents produce time/effort estimates for agentic work** — Agent execution is non-linear, so day/week/story-point estimates give false control; encode order via prerequisites and surface pace via blockers and unknowns instead.
