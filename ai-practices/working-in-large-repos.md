# Working in Large Repos: Good vs. Bad AI / Agent-Engineering Practices

A reference for steering AI agents through large, legacy, and multi-module codebases — how to map them, scope context, and structure rule files so the agent finds what matters without drowning in noise.

## ✅ Good practices

- **Prefer granular, area-scoped rule files over one monolith** — Keep repo-wide always-true rules in the root file and move part-of-code rules into a nested `AGENTS.md`/`CLAUDE.md` (or `.cursor/rules` with a glob); each short file sits fully near the top of its own section, so the agent never has to reach rule 47 at the bottom of a 400-line doc.
- **Child instruction files inherit from root and add only local specifics** — A child file should open with an inheritance line and list only module-specific gotchas, copying nothing from root; this avoids duplicating monorepo conventions and prevents drift between copies (e.g. cloudflare/workers-sdk: root ends with an index of children, child opens "Wrangler-specific context only. See root AGENTS.md.").
- **Escalate context structure on real signals, not repo size** — Add per-module `AGENTS.md` or a `context/` dir only when triggered by an observed agent failure or a real ownership boundary (root past ~300 lines, the agent repeatedly losing a module despite corrections, ever-growing reference passing, or a module gaining its own deploy/owner) — not on vague "my app is big" intuition.
- **Distribute shared multi-repo context from one source of truth** — Never copy-paste universal rules across repos; keep them in a dedicated company/ai-toolkit repo and distribute via an npm package, a CLI fetching over API, or an MCP server (least to most complexity) so copies stay aligned automatically.
- **Wide Scan then Deep Focus** — Run a cheap, deterministic broad survey with CLI tools first (a cheaper model suffices) to discover which areas are critical, active, coupled, and sensitive, then spend context budget on deep reading of a single chosen flow; this keeps each phase focused and stops analysis from sprawling across half the repo.
- **Treat the directory tree as a hypothesis, not the truth** — Folder layout is a static snapshot that hides project dynamics (what changes weekly, what breaks together, what is dead vs business-critical), so verify it against evidence; a search for "payments" finds the word but not whether payments are driven by webhooks, jobs, the admin panel, or feature flags.
- **Mine git history as cross-cutting narrative, not just per-line blame** — Top-changed folders (de-noised, split by quarter) distinguish a permanent core from a seasonal campaign, and co-change pairs/triples reveal coupling no branch view or `git blame` can surface — answering "where in this legacy must I be careful?".
- **Capture contributor context to recover knowledge that lives in people's heads** — Code can be read but decisions and edge cases often can't; build a contributor map (filtering out bots and AI-agent commits, grouped by theme) as an entry point to read PRs and ask the right person before repeating a change someone already attempted a year ago — not as a list of authorities.
- **Identify hotspots as complexity × change-frequency, not raw ugliness** — A frozen dense file (complexity alone) or a trivial config touched often (activity alone) both mislead; only their product flags code that is hard to understand *and* constantly changed, where every edit is costly and risky.
- **Confirm scan scope with the human before analyzing** — Propose the detected source dirs and exclude dist/coverage/fixtures/lockfiles before scanning, so generated artifacts don't bury the real churn signal that no one writes by hand, and the user can correct missed workspaces up front.
- **Use git history to date the artifact and diff only what changed since** — Pin a LAST_TOUCH commit with `git log --follow`, then run a targeted `git diff LAST_TOUCH..HEAD` on manifest/lint/test/CI files (the scripts block being the most common source of staleness) to focus re-inspection on what actually changed rather than re-discovering everything.
- **Scaffold the file system of record up front** — Establish a fixed foundation directory structure first (e.g. `/10x-init` creates `context/foundation/lessons.md` and `docs/reference/contract-surfaces.md`) so the rest of the workflow has stable, predictable paths to write to and agents don't invent ad-hoc locations.
- **Classify proposed changes by blast radius (easy/medium/hard)** — Tier edits by risk so it is clear which are cosmetic and which silently break contracts other components grep for: easy = trigger phrases/formatting (nothing critical breaks); hard = allowed-tools, output filenames, status values that break downstream skills relying on exact names.

## ❌ Anti-patterns

- **Scaffolding a greenfield bootstrapper over an existing project** — Running a greenfield bootstrapper on an existing repo causes scaffold collisions and proliferating sibling files; brownfield repos need an audit/health-check path instead (e.g. `/10x-health-check`, which audits dependencies, test runner, and CI, then emits an agent-readiness verdict).
- **Building per-folder instruction files before the project needs them** — Pre-emptively scattering instruction/context files across every module pays a maintenance cost for structure that yields nothing, and slow-changing files seeded early end up holding fast-changing facts that rot; MVPs with a few modules should stay on rung one as long as possible.
