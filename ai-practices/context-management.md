# Context Management: Good Practices vs. Anti-Patterns

Practical guidance for managing what enters, persists in, and survives an AI agent's context window. The effective context window is a finite attention budget — quality degrades long before the nominal limit — so the goal is to load only what each step needs, keep it lean, and persist the rest to durable files.

## ✅ Good practices

- **Treat the system prompt as a map, not an encyclopedia** — Keep it a lean, generalized "table of contents" of project-wide conventions and pointers the agent navigates and progressively expands, rather than front-loading every fact; this preserves attention for the task.
- **Keep the system prompt generalized; load specific knowledge via tools on demand** — Hold generalized interaction rules in the system prompt and pull documents in only when needed, so irrelevant content never adds permanent noise. The same tool attaches to many agents, so its description belongs in the system prompt only as a flexible, generalized role.
- **Keep the system prompt static to preserve the prompt cache** — Caching is the priority for latency (TTFT) and cost; even injecting the current date/time invalidates the cache and can critically hurt whole-system performance.
- **Inject dynamic state into user messages, not the system prompt** — Put frequently-changing data (git branch, open files, cursor position, environment) into the latest user message with xml-like tags separating it from the user's query, preserving a high cache hit rate while still giving the agent fresh state. A user message can legitimately contain more than the user's literal words.
- **Store the content map in external files, not the system prompt** — Detailed per-tool usage instructions backfire as they grow complex; keep the content map in external files the agent reads on demand.
- **Chunk inputs to focus attention even when they fit the window** — Narrowing scope per request raises accuracy and enables smaller, cheaper, faster models, even when the whole document would fit.
- **Reduce per-call context via decomposition and subagents** — Limiting how much is loaded at once preserves instruction adherence and retrieval accuracy; decompose tasks and delegate to subagents to keep each step lean.
- **Make subagent reports short and citation-bearing, not file dumps** — Cap subagent returns at ~200 words of facts with path:line citations so the main agent's context stays lean and verifiable; subagents are cheaper than the user's attention.
- **Probe the codebase with subagents instead of asking the user** — The codebase is a more reliable source of truth than user memory; fan out isolated probes that each return a short present/absent/partial verdict with file:line citations to keep the main context small.
- **Prefer compaction plus a filesystem over a sliding context window** — Truncating oldest messages breaks the cache and loses information; keep the thread, summarize near the limit, and spill full content to files the agent can re-read. Start compression early (~30% of the limit).
- **Persist long sessions through context compression** — Treating a main session as effectively infinite is viable only if compression (e.g. Observational Memory) keeps working context bounded as it accumulates.
- **Compress conversation history into a concise running log** — Observational Memory (Observer/Reflector) regenerates a precise log past a token threshold and seals/drops old messages, replacing semantic search and graphs (94.87% on LongMemEval with gpt-5-mini).
- **Spill large tool outputs to files and load only relevant fragments** — Big tool results (web search, long page fetches) bloat context; cap output, write the overflow to a searchable file, and let the agent (or a subagent) read just the parts it needs.
- **Use a filesystem as shared memory for inter-agent communication** — A common file store lets multiple agents exchange information and operate over long time horizons while context is compressed per session.
- **Read file fragments to learn structure rather than loading everything** — Loading 150k lines is infeasible and error-prone; read samples to understand the schema, then navigate the rest with code.
- **Use progressive disclosure plus sandboxed code execution to keep data out of context** — Load only meta-tools (list servers/tools, get schema, execute code) and discover the rest on demand; large datasets live as code variables that never enter the window.
- **Use Code Mode to keep tool definitions out of context for tool-heavy agents** — When the agent invokes tools by writing and running code, tool definitions need not be preloaded, saving tokens; the setting is per-agent so few-tool specialists can still load eagerly.
- **Account for MCP tool-definition token cost; prefer Code Mode at scale** — Each connected MCP server injects all its tool schemas into context (potentially over a million tokens), pushing earlier conversation out; code-execution patterns expose only `search()`/`execute()` at ~1000 tokens regardless of API size, and avoid env-var leaks via prompt injection.
- **Use progressive disclosure to keep skills cheap on the context budget** — Load only ~100 tokens of metadata per skill until activation, then ~5000 tokens of instructions, then unlimited resources on demand — so twenty skills cost ~2000 tokens, not ~100,000.
- **Keep SKILL.md lean and offload detail to reference/script files** — Cap the main file (~500 lines) and put extra docs in `references/` and code in `scripts/`; detail loads only when needed and script bodies never enter context, only their output.
- **Encode quality rules and named anti-patterns as on-demand reference files** — Keep locator hierarchies, isolation/cleanup rules, and named anti-pattern checklists in reference files loaded only when the skill runs, constraining the agent without bloating context between runs.
- **Separate behavior (SKILL.md) from data contracts (references/)** — Keep schemas, templates, and registries in companion files so the behavioral prompt stays focused and contracts can be versioned independently.
- **Keep the root instruction file a lean table of contents, not an encyclopedia** — Hold only project-wide, always-needed conventions and pointers in the root AGENTS.md/CLAUDE.md; a lean root (e.g. ~82 lines) preserves the limited attention budget for the actual task.
- **Separate context into three kinds with distinct homes** — Route conventions to AGENTS.md/rules, references (PRD, plans, research, decisions) to a `context/` system-of-record, and repeatable procedures to skills/prompts, keeping slow- and fast-changing material apart and loadable just-in-time.
- **Pull deep context just-in-time rather than holding it permanently** — Store PRDs, plans, and research in a `context/` layer the agent reads only when a task needs it (often via an explicit file reference), keeping the always-loaded surface small.
- **Treat the effective context window as a finite attention budget** — Advertised million-token windows mislead; strong models hold quality to ~250k tokens and cheaper ones degrade near 100k, so loading more permanent context measurably lowers quality and raises cost (one study: 20%+ drop in task success).
- **Order onboarding context by leverage, critical rules first** — Models show U-shaped ("lost in the middle") attention, so put the highest-stakes rules and most-used commands in the first third; a partial or truncated read still absorbs what prevents the worst mistakes.
- **Run from the module or pass an explicit reference; don't rely on lazy loading** — When critical rules live in a nested file, lazy auto-loading risks the agent skipping them; launch from inside the module or explicitly reference the file to deterministically load the full parent chain.
- **Schedule periodic context review and exclude fast-changing facts** — Module-owned files rot into harmful stale instructions; run periodic reviews and keep volatile facts out of slow-changing files to slow degradation.
- **Practice context hygiene before restructuring** — Before building new architecture, cut non-essential rules to free room for the ones that actually prevent repeated mistakes — a cheap first step before any structural change.
- **Enforce a hard length budget with pass/fail gates** — A concrete word budget (e.g. 200-400 at repo level, 120-250 per directory) with under/over diagnostics keeps artifacts lean; padding is a defect, not a style choice.
- **Reference exemplar/canonical files instead of inlining content** — Point the agent at the pattern file (`@src/.../user.service.ts`, `@package.json`) rather than pasting code or config; this keeps rules short and stops examples going stale in two places. If a rule duplicates the README/style guide/linter, delete it and reference the canonical source.
- **Onboard the agent with project rules before commanding it** — A fresh session never knows your team's conventions; encoding them up front shifts control before the session starts and shrinks the agent's room to wander, instead of forcing you to correct every diff.
- **Treat the user prompt as only the tip of the context iceberg** — System prompt, auto-memory, CLAUDE.md/AGENTS.md, skills, and commands all shape interpretation; even an empty repo consumes ~22k tokens, so place rules where they actually steer behavior.
- **Treat auto-memory as fallible, not as source of truth** — Cross-session memory can name functions/files/flags that were renamed, removed, or never merged ("memory says X exists is not the same as X exists now"); review it after important sessions and keep team-binding rules in versioned files.
- **Maintain one source of truth with thin tool-specific layers** — Keep AGENTS.md as the source of truth and import it from a thin CLAUDE.md / copilot-instructions.md (or symlink), rather than maintaining multiple drifting copies.
- **Persist artifacts to files rather than relying on conversational memory** — File-based contracts (PRD, tech-stack, plans, verification reports) survive context compaction and fresh sessions: "history is fragile, a file in the repo is durable." Store change.md, plan.md, plan-brief.md, and a Progress section in the repo so work is resumable and hand-off-able.
- **Persist scan results into versioned context artifacts** — Write each session's findings to agent-friendly markdown in the repo (e.g. `context/map/`), building durable project memory the next session starts from instead of losing it when the conversation ends.
- **Write context for the agent, not an essay for humans** — Use short sections, concrete lists, command-evidence traces, explicit "unknowns", and clearly marked caution zones; this is more useful and compaction-resistant than prose or a chat dump.
- **Use durable on-disk files as the memory of a multi-session rollout** — Persist state in files (test-plan.md, change.md, research.md) so a fresh session re-reads the disk and suggests the next step; work survives `/clear` and compaction without the human remembering where they stopped.
- **Provide a lightweight status command to resume long rollouts** — A `--status` view (phases closed/in-progress, next command) lets a developer return after weeks without re-reading chat history.
- **Persist resumable checkpoint state in the artifact frontmatter** — Writing current_phase, phases_completed, and status into the output file lets a long multi-step session survive interruption and resume from the next unfinished phase without re-running completed work.
- **Summarize completed phases on resume instead of replaying them** — Restore context with a 1-2 sentence summary per finished phase; replaying answered steps wastes the user's time and risks contradicting earlier load-bearing decisions.
- **Use checkbox progress markers as the single source of truth** — Drive a multi-step process from in-document checkboxes rather than a separate state sidecar, avoiding divergence between state file and artifact and simplifying resume.
- **Run work in small, checkable steps with a clean context per stage** — Cramming strategy, research, planning, and implementation into one long session loses half the decisions; small steps with a fresh context at each stage preserve fidelity.
- **On drift symptoms, start a fresh session instead of re-prompting** — When the agent loops on fixes, mangles file names, or loses prior decisions, reload clean state from disk (plan.md + latest Progress) rather than compounding the polluted context.
- **Name and counter context drift as a predictable system property** — Drift is a cascade of small wrong decisions producing internally consistent code that solves the wrong problem; recognizing it as architectural lets a research→plan→implement workflow with on-disk plans and phased commits mitigate it deliberately.
- **Default each autonomous action to a fresh session to avoid context pollution** — Start a new session per trigger so unrelated background events don't contaminate the main context; share the main session only when an event genuinely needs its context.
- **Isolate sessions and persist conversation history per session key** — Concurrent users need independent, keyed conversation state so context never leaks between sessions and the thread is maintained across turns.
- **Use git worktrees to isolate parallel agent sessions** — Each worktree gives an agent its own working directory, branch, HEAD, and index so concurrent agents edit files without corrupting each other's git state.
- **Prevent conflicts via resource ownership, read-only scopes, and session isolation** — Avoid concurrency conflicts by design — assign resource ownership, grant read-only permissions, and isolate context per session — rather than resolving them after the fact.
- **Detect write conflicts with read-time checksums before committing** — Because agents read before modifying, compare a checksum or per-line hashes at write time to detect intervening edits, borrowing version-control discipline for shared agent state.
- **Decouple external context from agent logic** — Shared knowledge/memory should describe data, not encode which agent does what and when; binding orchestration logic into documents tightly couples the system and blocks human collaboration on those docs.
- **Give the agent an upfront map/index of available knowledge resources** — Agents "don't know what they know"; an auto-generated `_index.md` or discovery hints lets the model navigate resources it would otherwise miss, preventing false "I can't do this" conclusions and enabling deliberate search.
- **Build knowledge bases for agents instead of connecting agents to existing docs** — Designing the knowledge layer for the agent sidesteps retrieval entirely — each agent is given exactly what it needs and never has to "find" information.
- **Let agents navigate by following links embedded in document content** — Like coding agents traversing source (grep one file, follow imports), making documents reference each other lets an agent walk only the relevant slice of a large KB step by step instead of loading everything.
- **Make links explicit and self-describing rather than implicit references** — Cross-note links and descriptive paths let an agent traverse and expand context; shortened or label-less links are effectively invisible and break the graph it needs.
- **Write notes assuming the reader has zero prior context** — Humans omit what they know (project names, who people are, what "last conversation" means), but an agent cannot infer it; these gaps directly degrade retrieval and reasoning.
- **Always cite source and location for retrieved chunks** — Telling the model which document a fragment came from improves grounding and lets the UI surface links and quotes beside the answer, helping users catch mistakes.
- **Match chunking strategy and metadata to how the agent will retrieve** — Decide "how to create documents?" together with "how will the agent reach them?"; contextual/thematic splitting enables keyword and tag metadata that naive character splits cannot.
- **Present documents as images, not just extracted text, when structure matters** — Charts, screenshots, and embedded images are lost in text-only passing; rendering as an image lets vision models reason over structure and can compress even textual content far more efficiently.
- **Version documents instead of overwriting, and signal the latest** — Overwriting destroys information the agent may still need; when new versions exist, tell the agent which is current and how to reach it.
- **Preserve context and metadata when persisting information from a session** — Facts clear in conversation lose their referent once written to an independent note; record metadata (source, date) so a note about "Anna" is not confused with another, and to support later queries.
- **Detect duplicated knowledge with periodic cheap-model scans** — Even with good architecture, knowledge ends up in multiple places; run small cheap models over recently-modified data to flag likely duplicates rather than trying to prevent it entirely.
- **Deduplicate and cache repeated inputs before paying for model calls** — When many inputs are identical or near-identical, cache the model's answers yourself and collapse duplicates to cut cost dramatically at scale.
- **Minimize model output, not just input** — Output tokens cost more than input, so a verbose response is disproportionately expensive even with large input; constrain the model to the smallest sufficient result.
- **Keep every agent interaction as short as possible** — Even with compression logic, shorter interactions improve task success and lower cost, since you pay per token consumed; brevity is a direct lever on accuracy and spend.
- **Estimate tokens before the call and refine from API usage afterward** — Pre-estimate via chars/4 plus a ~20% buffer to trigger compression early, then use the API's reported token counts to correct the estimate and tune max_tokens next time.
- **Optimize tool responses for token efficiency by default** — Tool outputs flow back into context, so trim them at the source to keep agents cheaper and leave more room for reasoning.
- **Compact test output for the agent to save tokens and reduce noise** — Verbose test output floods context with noise; a compact failures-only reporter (e.g. `AI_AGENT=1` with Vitest 4.1+) keeps signal high and context lean.
- **Consider loading documents directly into context instead of building search** — If the relevant documents fit, reading them straight into the conversation can be fully sufficient and avoids the cost and complexity of a retrieval system.
- **Prune known-irrelevant context up front** — Exclude documentation areas you have already decided not to address (e.g. skip "Gmail settings management" when saving the Gmail docs) to keep the working context focused on the surface you intend to build.
- **Ground the model in fresh API docs and SDK source before designing tools** — Models answer "from memory" on well-known APIs; saving the docs as Markdown and pulling the official SDK gives current, verifiable context both model and human can read.
- **Feed current docs into context because the model's protocol knowledge may be stale** — For fast-evolving standards like MCP, training knowledge is likely outdated, so load the authoritative README/manual and re-remind the agent of it frequently to avoid wrong implementations.
- **Have a coding agent study the official SDK and write a reusable integration note** — Distill the SDK's interface into a persistent note instead of re-deriving integration details each time, giving a reference for every future feature touching that system.
- **Document tool usage in context so the agent does not burn tokens on trial-and-error** — If a custom CLI's commands and conventions aren't in a skill, rules, or prompt, the model rediscovers usage by trial and error, wasting tokens and producing flaky behavior.
- **Read source artifacts fully before analyzing, no truncation** — Analysis quality depends on complete context; read the SKILL.md and every referenced doc in full before producing any report, since partial reads lead to wrong conclusions.
- **Separate the model's base knowledge from external context in instructions** — Tell the agent that facts about itself, the user, and the environment come from injected context, not its training, so it doesn't confidently assert things it cannot yet know and recalls progressively.
- **Enrich context to increase useful output diversity** — Model behavior is strongly conditioned on preceding content, so giving an agent environment state and long-term memory raises response diversity and relevance far more than sampling parameters do.
- **Ground tools in environmental context for proactive, accurate use** — Supplying metadata about surroundings (current date, available tools/agents, assigned knowledge structures) lets agents use tools more effectively and act proactively.
- **Synthesize and paraphrase gathered context to cut repetition and emphasize key behavior** — After collecting what a model lacks, deliberately synthesize and paraphrase to avoid redundancy and weight chosen behaviors; raw accumulation degrades instruction quality.
- **Inject reusable skills as predefined instructions rather than re-prompting** — Predefined skill instructions injected by user action or model decision are a high-leverage personalization mechanism; add search and grouping once they grow numerous.
- **Use frontmatter fields to let users steer agent behavior per-document** — When a model can't reliably judge what to add, a lightweight per-file directive gives targeted human guidance without a heavy configuration system.
- **Prefer simple documents over elaborate long-term memory until proven necessary** — Complex memory categorization creates ambiguous, dynamic decisions agents organize poorly; simple, maintainable document structures are often sufficient and more robust.
- **Don't deduplicate cross-references aggressively in linked notes** — When an agent retrieves only a fragment, a relationship mentioned just once elsewhere may be absent from that fragment, so it never discovers the link; some repetition of associations aids retrieval.
- **Use cloneable templates and reference-by-name to avoid rewriting whole artifacts** — Have the agent clone a JSON/HTML template, edit only the changed fragments, and pass the file by reference rather than re-emitting the whole document, saving tokens and reducing transcription errors.
- **Make every tool call and file peek deliberate** — Treat each command and each look into a repo area as a purposeful action to obtain a specific, repeatable slice, avoiding burning context budget on random exploration.
- **Feed the agent a fixed-structure map as an input contract** — Passing a prior project map with stable sections (risk zones, entry points, constraints) directs where the agent spends tokens instead of letting it read randomly.
- **Treat a stale map as a prior to verify, not as ground truth** — A map is a time-snapshot of structure, not current behavior; feed it as a prior the agent must confirm so an outdated assumption is never silently rewritten as fact.
- **Scale context preparation with task risk, not uniformly** — Routine CRUD needs only a light plan, but slices with unique business logic require front-loaded research before planning; match prep depth to risk to avoid under- and over-engineering.
- **Persist plan and research as files on disk, not in long conversation** — A durable on-disk document (e.g. research.md at a fixed path pulled via `@research.md`) is a stable anchor that survives session resets and that downstream steps compare against.
- **Keep PRD, tech-stack, roadmap, backlog, and plan as separate zoom levels** — Each artifact answers a distinct question (what/for whom, with what, in what order, what now, how exactly); mixing levels is a primary source of chaos that AI amplifies.
- **Separate the exploratory session artifact from the formal contract** — Keep shape-notes (a decision log) separate from prd.md (a structured contract) so each step does one job and the second is a faithful transcription, not a fresh invention.
- **A stable change-id binds roadmap item, folder, plan, and progress** — One operational identifier ties the roadmap item, change folder, plan, brief, implement command, and progress log together so the conversation doesn't fragment and there is one address for the change's memory.
- **Provide a short brief alongside the full plan for human review** — Long detailed plans are hard for humans to read, so a lightweight plan-brief.md enables fast assessment and hand-off without forcing reviewers through the full document.
- **Encode phase boundaries as explicit intent in a persistent change record** — Writing intent like "no refactor during exploration; the decision is made during planning" into the durable change file means every later session and skill inherits those guardrails.
- **Keep research-only changes as first-class artifacts** — A research-only change produces a report the next change consumes as a prior, so shared `context/` memory accrues reusable evidence instead of tangling research into an unrelated change folder.
- **Maintain a project test cookbook for reuse by future agents** — Record concrete patterns (test location, mocking policy, reference test, local command) after each phase so future agents don't re-guess the project's local testing culture.
- **Record the project's testing conventions in CLAUDE.md/AGENTS.md after introducing new infra** — Persist how the repo tests (including non-standard gates like mutation testing) so the agent stays aligned on later changes and transfers durable process knowledge across sessions.
- **Capture recurring findings as durable project rules (lessons)** — Some findings signal a missing local rule, not a one-off bug; record them (e.g. to `context/foundation/lessons.md`) so future planning/implementation runs load them as prior context and the same findings stop recurring.
- **Keep deterministic hooks as a backstop for unreliable model memory** — Instructions in CLAUDE.md can be compressed, ignored in long context, or forgotten, whereas a hook runs outside the model and always fires, making hooks the durable enforcement layer.
- **Maintain a load-bearing-names registry to keep contracts consistent** — A registry of contract surfaces plus a lessons file preserves critical naming and recurring rules across the workflow so agents stay aligned over time.
- **Make the schema reference the single source of truth, re-checked at every write** — When multiple skills produce/consume a shared artifact, pin its shape to one schema doc and re-validate at each checkpoint to prevent structural drift.
- **Persist findings to a durable context artifact for downstream skills** — Write the verdict to a system-of-record file (e.g. `context/foundation/health-check.md`) so findings survive the conversation and feed later steps, rather than staying trapped in chat.
- **Cross-reference complementary reports instead of duplicating analysis** — Link to an upstream report's findings (and reinforce compounding risks) rather than redoing the work, producing a richer combined picture than two siloed reports.

## ❌ Anti-patterns

- **Modifying the system prompt mid-session breaks the prompt cache** — Tool definitions sit below the system prompt in the context window, so any edit to it evicts the tool definitions from cache, hurting latency and cost.
- **Putting out-of-order state in the system prompt scrambles event chronology** — Adding task-progress or session summaries to the system prompt can present older facts than later messages, disturbing the chronology the model reasons over.
- **Don't inject a static directory tree or category map unless the structure is truly fixed** — Showing the full structure helps only when it stays constant during the interaction; otherwise it misleads the model or wastes context most of the time.
- **Large added context degrades instruction-following** — Loading big documents shifts attention to the new content, so the model silently drops or ignores parts of its system instructions (formatting, behavioral rules) even in the latest models.
- **Sketchy resource descriptions block deliberate retrieval** — Without an upfront map of what resources exist, an agent can only see fragmentary descriptions, leading to poor query formulation and incomplete retrieval.
- **Long conversations degrade model quality even before the window fills** — More input tokens lower precision (lost-in-the-middle, semantic distractors), so dragging out a single session is itself a failure mode that bigger windows do not fix — it is an architectural property, not a training gap.
- **Don't dump the whole repo into the model to understand it** — A large repo rarely fits once system prompts, rules, tool defs, and history are subtracted; even a fitting slice yields a flat, priority-less summary and a false sense of understanding.
- **Avoid the random deep-read anti-pattern** — An unconstrained agent opens an important-looking file, then its imports, then neighbors, and a dozen tool calls later it was a dead adapter nobody uses — "lots of traffic, few decisions" that burns context for no decision.
- **Treat whole-repo packers like Repomix as one-off snapshots, not standing infrastructure** — Dumping an entire repo carries a high token cost at both generation and read time, so it suits one-off lookups but not a permanent contract layer.
- **Endlessly growing a single monolithic instruction file** — A monolithic file past ~300 lines crowds out the task, dilutes key rules among trivia, rots faster than anyone reviews it, and makes compliance hard to verify; Anthropic warns an overfilled CLAUDE.md makes Claude ignore real instructions lost in the noise.
- **Bloating the rules file with redundant docs degrades the agent** — Long files restating framework docs and patterns the model already learned fill context with noise; studies found doc-stuffed context files raised cost ~20-23% and worsened success, with harm coming specifically from redundancy.
- **Never duplicate a single rule across multiple files** — Duplicated instructions drift apart as fast as duplicated code; maintain one source of truth with thin tool-specific layers importing it, not three copies.
- **Lazy loading of nested context is unreliable** — When critical module rules live in a nested file, relying on auto-load risks the agent skipping them; starting from root loads only the root file, so run from inside the module or pass an explicit reference.
- **Concurrent writers to shared memory cause silent lost updates** — Multiple parallel instances of even one agent template can write the same memory note, and a later write silently clobbers an earlier one, losing information with no error.
- **Disallow deleting mid-conversation messages** — Removing messages from the middle corrupts history, degrades model behavior, and breaks context-management mechanisms (some APIs even reject a request whose last message isn't the user's); branch or revert to a point instead.
