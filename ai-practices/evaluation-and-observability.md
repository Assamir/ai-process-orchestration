# Evaluation and Observability

Practices for measuring whether an AI/agent system actually works and for making its behavior debuggable. The recurring theme: judge models and prompts empirically on your own tasks, instrument every interaction, and never trust fluent output without verification.

## ✅ Good practices

- **Choose models empirically on your own representative tasks** — "Which model is best?" is the wrong question; build a personal verification suite of hard/important tasks, try 2-3 candidates on your actual workload, and weigh stability and scope rather than trusting a single passing benchmark or leaderboard rank. Published benchmarks are "one task, one benchmark."
- **Pick a search/tool provider by your task category, not overall rank** — top agentic-search providers are often statistically indistinguishable overall, yet diverge sharply per query type, so select on the category matching your real work and test a working provider on a task like yours before swapping it.
- **Use small local models as a litmus test for tool design** — if a small local model can correctly drive your MCP tools, that is usually a signal the tools are well designed, and it makes token cost effectively free during testing (swap the API base URL/model name).
- **Centralize observability hooks on every LLM call and tool invocation** — you cannot debug emergent behavior unless every request, tool call, and model response is captured, so design the architecture up front to make this instrumentation centralized rather than scattered.
- **Log the full session context, not just the bare event** — capture user id, session id, agent ids, and request metadata (a Session/Trace/Span/Generation hierarchy) so traces are actionable for debugging and cost attribution; also record how system instructions evolve over time.
- **Snapshot model/agent configuration per run** — store the model config and agent "revision" at request time so past agent activity can be reconstructed and debugged when investigating user issues.
- **Emit state-change events for observability, evaluation, and guardrails** — send an event (e.g. via SSE) on every state change (task creation, artifact modification, tool call) to get a hookable stream for live visualization, observation, evaluation, and inserting guardrails.
- **Start with a minimal observability integration and let gaps surface** — a minimal integration immediately reveals what is missing and where logs are too sparse or too noisy, a faster path to the right instrumentation than designing it all up front.
- **Debug agents by observing execution steps, not by reading code** — failures from dynamically loaded context and overlapping descriptions are nearly invisible in source; tracing the agent's actual steps reveals root causes that code inspection cannot.
- **Evaluate LLM behavior against thresholds, not exact-match assertions** — natural-language outputs have many valid surface forms, so score how well a result satisfies criteria on a 0-1 scale and require a "good enough" threshold instead of demanding a 100% match.
- **Run evals both offline (pre-release) and online (in production)** — offline evals in dev/CI predict regressions before shipping prompt changes; online evals score live interactions and catch violations that built-in guardrails missed.
- **Version system prompts with their run statistics, not just in Git** — Git rolls back prompt text but agentic systems also need the linked run history and cost/quality stats per version; if bidirectional storage isn't feasible, one-way sync into the platform still preserves current-state statistics even though you lose easy version switching.
- **Drive model/config selection with datasets and evals** — for tasks where the agent autonomously decides placement and wording, build eval datasets to reveal which model and configuration actually performs best rather than guessing, and use evals to gate model upgrades in complex systems where a smaller variant may only "seem" more capable.
- **Use an offline dev eval harness seeded from the tool's own repo** — a development-time eval tool (e.g. Promptfoo) surfaces problems early; pulling the eval tool's repo into context unlocks its newest assertion types and raises answer quality versus relying on memory.
- **Run scenario evals with a minimal system prompt to test the interface itself** — evaluating multi-step scenarios with almost no system message isolates whether tool descriptions and schemas alone are good enough; if the bare interface works, specialization only improves it.
- **Benchmark multiple models against the same eval suite** — an eval suite lets you cheaply compare candidates on cost, speed, steps, tone, and completeness, and decide whether downgrading to a smaller/local model is viable for the task.
- **Validate the artifact on a fresh run, not by assumption** — onboarding docs (and similar outputs) only prove their value when a fresh agent session actually uses them to do a real task, so build that validation reminder into the workflow to close the feedback loop.
- **Use an LLM-as-judge to continuously observe system effectiveness** — an observer agent can cyclically verify information flows, detect dead/unavailable data sources, and flag processes that have lost value, marking them for automatic or human-confirmed removal.
- **Audit the actual code to ground state, instead of asking "what do you have?"** — parallel subagents inspecting each layer (frontend, backend, data, auth, deploy, observability) and returning present/partial/absent verdicts with an evidence-file reference make the code itself the source of truth, avoiding reliance on faulty human memory.
- **Treat the structure guarantee as not a value guarantee in structured output** — "guardrailed/deterministic" JSON only constrains shape; a perfectly-shaped object can carry wrong values no schema check catches, so validation must go beyond conformance.
- **Score input quality and warn on thin material before generating** — a heuristic check on inputs (e.g. a 4-signal scan for concrete structural markers) lets the agent refuse to produce a polished-but-empty artifact and instead send the user back to do the real work.
- **Emit a structured, file-based verification report with per-phase status** — a durable post-execution report with explicit passed/warned/failed/skipped status and a short reason per phase gives an auditable record and tells the user exactly where to start investigating.
- **Make each finding state what is wrong, why it matters, and what to do** — a fixed finding schema (problem, agent-workflow impact, concrete fix, effort estimate) makes reports comparable and actionable and forces the tool to justify why an issue matters.
- **Demand explicit evidence / inference / unknown separation in reports** — forcing the agent to mark which steps are confirmed in code versus assumed makes output auditable and surfaces corrected assumptions instead of letting guesses pass as facts.
- **Record tool blind spots explicitly as unknowns** — static import graphs and git history miss runtime coupling (DI, dynamic import, feature flags, reflection, codegen, webhooks), so mark these as "unknown" rather than implying "no dependency"; a missing dependency graph is an unknown, not no-connections.
- **Read git co-change as a windowed signal, separating pairs from hubs** — co-change reveals hidden contracts but lies if read literally: a coupled pair is different risk than a hub coupled to ten files, and mass refactors spuriously link files, so rank risk from both the static graph and co-change and treat contradictions as questions.
- **Triage the symptom to choose the right evidence source first** — matching symptom type to source order avoids wasted investigation; a "200 but nothing changed" signals a silently swallowed error and points to monitoring (e.g. Sentry) before logs or code.
- **Treat a vision model's UI diagnosis as a hypothesis, not evidence** — VLMs have spatial bias near screen edges and can both hallucinate problems (false positives) and miss real ones (the more dangerous false negatives), so confirm their open-ended answers.
- **Tune monitoring capture levels as traffic grows** — capturing all console warnings/errors maximizes early visibility but consumes the same event quota as real exceptions, so at scale narrow levels (e.g. to `error` only) or report selected cases so genuine outages aren't buried in noise or over budget.
- **Let a later pipeline stage refine, not just consume, earlier findings** — treating prior analysis as improvable input rather than fixed truth lets a fresh re-read correct earlier mistakes and surface problems the first pass only hinted at.
- **Frame gaps by context: don't penalize what isn't yet relevant** — separating "fix now" findings from "expected, coming later" ones keeps the verdict honest and proportionate and avoids alarming users about things outside the current scope.
- **Output decision-ready Markdown tables before generating diagrams** — return a short Markdown answer (key observations plus a table) first; it is more legible and easier to refine than a dense diagram, and reserve SVG/graph rendering for the one subgraph that already has decision value.
- **Match retrieval strategy to project requirements** — direct context loading, text-file navigation, hybrid search, and graph indexing trade accuracy against cost, complexity, and processing time, so there is no universal solution; choose per project, and treat vector-only search as deprecated, layered only on top of full-text/grep when lexical search proves insufficient.
- **Validate new releases by testing yourself, not by following hype** — marketing noise makes real usefulness hard to judge, so hands-on testing grounded in your own experience is a far better signal of whether to invest attention.
- **Don't pre-emptively research every unknown** — research only earns its cost when it unblocks a decision; local unknowns can be resolved by the planning agent reading the code, so closing every unknown up front is wasted time.

## ❌ Anti-patterns

- **Treating the LLM domain expert as a source of truth** — domain-expert hallucinations are insidious because you ask about things you don't know, so a confident answer reads as knowledge even when fabricated; the model confabulates precise constants, thresholds, third-party field names, recent changes, and niche regulated domains, so cross-check any load-bearing claim before it reaches a class name or schema.
- **Trusting multimodal output because most of it looks right** — generated images (and audio) contain subtle "glitches" that are hard to catch precisely because 95% of the detail is correct, so treating image output as trustworthy is a trap.
- **Relying on an LLM as the sole safeguard against malicious content** — an LLM risk classifier can itself be deceived by an instruction crafted to bypass it, so enforce validation, moderation, and access controls in deterministic code and let the agent receive only error info and guidance.
- **Picking the cheapest model on per-token price alone** — a weaker model that needs many more steps or extra tool calls can cost more overall and exhaust strict rate limits faster (e.g. GPT-4.1-nano proved less cost-effective than GPT-4.1-mini).
- **Designing only for hard stops, ignoring silent failures** — missing information or a missing decision may not halt an agent but instead make it skip steps and produce a plausible-but-incomplete result, so observability must catch errors that aren't immediately visible.
- **Assuming the model will search your documents over its own knowledge** — if an answer exists in parametric knowledge the agent may skip searching your sources, returning stale or contextually-wrong answers, and may not realize your terms mean something specific in your domain.
- **Trusting evals that flatter the new version instead of guarding the contract** — auto-generated evals tend to test what the new version happens to do well rather than what it should do, so they pass while real quality regresses; design evals around the contract's limits and edge cases. A good eval is harder than a good skill.
- **Chasing mutation score to 100%** — treating mutation score like coverage drives writing tests that pin arbitrary details, recreating implementation-mirror fragility; the real product is the list of live mutants, each triaged individually (real gap, equivalent mutant, or not-worth-pinning).
- **Reading change counts naively** — a high change count can mean a product core or a place where one hard bug was patched all year, so treating the ranking as truth conflates importance with brokenness; co-change is also history, so verify the files still exist before relying on them.
