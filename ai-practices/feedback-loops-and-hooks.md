# Feedback Loops & Hooks: Good vs. Bad Practices

Agents get reliable not by being smarter but by being wired into tight loops — verifying their own work, reacting to deterministic signals, and capturing what they learn. This reference distills good practices and anti-patterns for designing those feedback loops and hooks.

## ✅ Good practices

- **Verify after each batch of actions instead of blindly executing a plan** — When an agent acts on a possibly-wrong perception (e.g. a misread image), re-fetch state and check it against the target after a few actions to catch errors early and avoid wasted or destructive operations.
- **Iterate on output using precise downstream feedback** — A verifier that states exactly what is missing turns a one-shot guess into a convergent loop; start with a smaller set and expand from feedback rather than dumping everything up front.
- **Make token-counting an explicit pre-submission step with a conservative ratio** — Hard token limits cause silent rejection, so count tokens as a discrete verification step before sending and use a conservative conversion estimate to prevent wasted round-trips.
- **Adopt a reactive, feedback-driven loop instead of fully understanding an API upfront** — When a tool returns precise errors, send a best-effort attempt and correct iteratively rather than decoding the entire spec first; this also covers live data sources where you retry on "missing/wrong" feedback instead of assuming an item does not exist.
- **Use a separate, isolated classifier prompt as a prompt-injection barrier** — Run user input through an independent model call that returns a secret safe/unsafe token, then check it in code, keeping the decision phrases hidden from attackers and hard to bypass.
- **Make agent outputs deterministic triggers for downstream automation** — Treat an agent's observable side effect (e.g. a "Bug Report" label) as an event that chains further agents or deterministic actions, composing pipelines instead of one monolithic agent.
- **Drive proactive behavior via a heartbeat that reads a tasks file** — A periodic message asking the agent to read a tasks file and act only when needed enables proactive assistance, while letting the agent silently skip when nothing requires attention.
- **Persist per-domain learnings so the agent doesn't rediscover a site every run** — Store site structure, notes, and reusable scripts in an instructions directory so the agent reuses knowledge across sessions; on errors, have it screenshot and save notes that improve the whole domain, not just the current run.
- **Use lifecycle hooks as active process guardrails, not just background side-effects** — Step/tool hooks (onStart, onStepFinish, beforeToolCall, beforeFinish) can verify state, track progress flags, and re-prompt the agent to finish required steps — actively enforcing process correctness, with sensible exceptions for step-limit or tool-error cases.
- **Insert human-in-the-loop checkpoints at hook boundaries** — Session/tool-start and before-finish hooks are natural places to require human confirmation of untrusted actions, supply missing info, or decide whether to escalate versus just report status.
- **Automate optimization with checklists and good-practice docs the coding agent can run** — Codify good practices and checklists into documents so a file-reading, terminal-capable agent can review the interface and tests, run and interpret evals, and propose improvements semi-autonomously.
- **Make tool invocations retryable by reusing the existing payload after failure** — When a tool call fails, let the agent modify and resend the existing payload (e.g. an email body) rather than regenerating it from scratch, strengthening error recovery and efficiency in the agent loop.
- **Only add a rule when it removes a repeatable agent error** — Calibrate every rule, skill, or hook against where the agent actually breaks conventions rather than adding it "just in case"; if the agent already meets the convention unaided, the rule only burns context.
- **Distinguish hooks (deterministic mechanical checks) from skills (invoked procedures)** — Use hooks for mechanical controls the agent must not skip (format on save, lint/test the changed file, end-of-turn command) and skills for repeatable but non-automatic workflows (deeper verification, post-session report, release checklist).
- **Capture recurring incidents as durable lessons, not a log of every mistake** — Record classes of errors that will recur in an append-only lessons file (Context/Problem/Rule/Applies-to); fix one-off oddities locally, but persist patterns that should change future framing, planning, or review.
- **Capture skill/agent preference corrections as durable rules for future runs** — When a skill or agent acts against your preferences (e.g. analyzing irrelevant parts of the repo), feed that observation back into persistent rules instead of correcting it ad hoc every session.
- **Automate quality checks via agent hooks instead of manual re-runs** — Wire a hook to run lint, types, and tests automatically at the right local moment so the edit-verify-fix loop closes without manually repeating it after every edit or waiting for CI.
- **Use the universal four-part hook model: trigger, matcher, handler, signal** — Every AI coding tool shares the same trigger (event) -> matcher (filter) -> handler (command/script) -> signal (exit code + stdout) architecture, so learning the pattern once makes hooks portable; only config format and event names differ.
- **Feed hook output back into agent context so it self-corrects** — When a blocking hook returns a clear error (missing type, unimported module, bad format), the agent sees it on its next turn and fixes trivial issues autonomously, removing a manual round-trip.
- **Standardize hook exit codes: 0 pass, 2 blocking-with-feedback, other non-blocking** — Correct exit-code semantics decide whether the agent gets actionable feedback or silently ignores a failure, so treat 2 as the must-react signal and reserve other non-zero codes for logged, non-blocking errors.
- **Budget hook latency against the agent loop it blocks** — A synchronous hook stalls the agent until it finishes, so a multi-second check across dozens of edits becomes a major drag; move slow checks to commit or push time, and treat an annoying hook as a signal to change its layer.
- **Prefer the most reliable hook combination (PostToolUse + command handler)** — Hook reliability varies by tool and event type; PostToolUse with a command handler fires most consistently across tools, whereas Stop hooks and some PreToolUse capture are unreliable.
- **Add a git pre-commit gate so non-agent changes are still checked** — Agent hooks only fire when the agent works, so a pre-commit gate on staged files catches manual edits, files changed outside the hook, and checks too slow to run per edit.
- **Know that PostToolUse fires once per tool call, not once per turn** — If the agent edits three files in one turn the hook runs three independent times, which is fine for fast checks but warrants a more restrictive matcher when the check is slow.
- **Use mutation testing as a separate quality gate for assertion strength** — Coverage only tells you a line ran; mutation score tells you whether a test would notice a change, exposing AI-written tests that leave code covered but undefended — a surviving mutant is concrete proof an assertion is blind.
- **Stop and revise the plan when a discovery changes the contract** — Adapt small mismatches in place, but when the agent finds the contract surface differs from the plan's assumptions, make it explicit by returning to the plan rather than patching with another prompt and amplifying drift.
- **Insert quality gates so agent output can't go straight to production** — Because agents produce large amounts of plausible-looking code fast, checkpoints (lint, typecheck, unit tests, a critical-flow test, PR review, pre-commit, CI) deliberately force tool/test/review signals before a change advances past "works locally."
- **Treat a working test runner as the agent's verification feedback loop** — Without an executable suite an agent cannot verify its own changes, so detecting the test runner and dry-running it to confirm tests actually execute is a top-tier readiness criterion, not a nice-to-have.

## ❌ Anti-patterns

- **Relying on the model to maintain a to-do/plan without programmatic support** — Without code enforcing updates, models forget to refresh the task list or only update it after finishing everything, defeating the attention-management purpose.
- **Expecting agents to auto-fix deep business-logic failures from a hook** — Hooks surface that a test failed, but for genuine logic errors the agent often guesses a superficial patch that only appears to succeed; recognize this boundary and escalate to a full change workflow instead of trusting the gut-feel fix.
- **Trusting hooks and code-level checks to catch failures that cross system boundaries** — Linters, typecheck, and unit-test hooks read source code and never start a server or query a database, so data lost along an auth -> API -> DB path passes every hook yet breaks the real user flow; only full end-to-end tests catch it.
