# Code Review & Verification of AI/Agent Output

AI agents trade reliability for adaptability, so their output must be verified rather than trusted. These practices cover how to keep a human accountable, anchor review to a plan, and prove that tests and structural claims actually hold.

## ✅ Good practices

- **Keep a human in the loop to verify results** — Agents are never 100% reliable, and verifying output is almost always cheaper than doing the task by hand; escalate ambiguous merges and conflict resolution to a person who has the context the agent lacks.
- **Supervise high-stakes outputs even when code does the math** — Reliable arithmetic in code does not guarantee the right data was loaded correctly; reports that drive important decisions must be human-verified, with AI used for support (transforms, classification, parallel analysis), not ownership.
- **Delegation lowers interaction, not responsibility** — Reducing in-the-loop control is fine only when accountability shifts to a later gate; the human still reviews each PR, owns the merge verdict (approve / request changes / block), and the standards meant to outlast one slice.
- **Require explicit accept/reject on agent suggestions** — Since perfect accuracy is unrealistic, force a human accept/reject decision (with an optional re-check prompt) so value is preserved while the cost of model errors stays bounded.
- **Keep the human as architect; AI as analyst** — Skills accelerate context-gathering and surface refactor candidates, but architectural decisions and deliverables must be ones the human can defend, not accepted on faith from a single prompt.
- **Review AI-generated evals for tests that can't fail** — AI may implement the evals, but a human must understand them well enough to catch logical flaws and tests detached from reality that "cannot not pass."
- **Treat behavior fixes as non-local: re-verify the whole system** — An instruction or tool-description change that fixes one case can silently break another, so a local fix is not the end of the work — prefer generalized rules and re-evaluate broadly.
- **Verify assertions by deliberately breaking the code (mutation check)** — A green report never proves an assertion catches anything; invert or weaken the protected behavior, confirm exactly the right test turns red, then revert without committing. The assertion is the most common weak point in LLM-written tests.
- **Review assertion provenance, not just the coverage bar** — Ask "where did this value come from?" for each assertion; if the answer is "that's what the function returns," the test mirrors the implementation and is not a real test.
- **Verify structural and numeric claims with a deterministic tool** — LLMs are weakest at exact counts and "only here" / "always via X" structural claims because they see tokens, not syntax nodes; confirm each such claim with a parser like ast-grep (and grep to confirm every zero). The more confident a number looks, the more it needs checking.
- **Independently verify the structural numbers a ranking rests on** — Confident counts are exactly where AI reports approximate; re-running ast-grep sharpened "about 40" into exact file:line facts (5 of 57 sub-stores, 150 not 145 imports) before a decision depended on them.
- **Demand evidence for every claim and label** — "A label without evidence is a guess": every finding must cite a file:line, command, or commit (entry points, imports, change history, dependency paths), and absence of a reference means no claim rather than an estimate — separating fact from interpretation.
- **Anchor review to the approved plan, not gut feel** — Reasonable-looking code can solve the wrong problem; ask "does this change deliver the plan per the requirements?" using plan.md as the record of intent, instead of intuitive diff-reading.
- **Run a plan-review gate before writing code** — Reviewing the plan (does it answer the task, is the end state concrete, are phases executable, are contract surfaces named) is the cheapest moment to course-correct — a bad plan costs minutes, a bad multi-file diff costs far more.
- **Give the agent a checkable success criterion, not a subjective goal** — A concrete, grep-verifiable acceptance test (e.g. "after refactor, grep for the package name returns only ACL/adapter files") forces the plan to optimize for something you can confirm in seconds.
- **Review with a multi-dimensional scorecard in risk order** — Read diffs in a fixed order — scope vs plan, then risk boundaries (auth, data writes, migrations, external APIs, env config), then local patterns, then tests — across six dimensions (Plan Adherence, Scope Discipline, Safety, Architecture, Pattern Consistency, Success Criteria) so dangerous deviations surface before cosmetic nitpicks.
- **Watch for silent scope creep in the diff** — Agents tend to solve a slightly bigger, "prettier" problem than asked; if a diff touches files outside the slice's intent, halt and require a justification you understand.
- **Verify AI reuses existing local patterns** — Agents often produce technically-correct but non-conforming solutions (new error formats, folder structures, helpers); check that code reuses existing helpers, names, and conventions rather than reinventing them.
- **Triage by severity AND impact — not "fix everything"** — Separate how serious a problem is from how far its fix reaches: a high-severity/low-impact validation gap can be fixed at once, while a low-severity/high-impact enum rename touching frontend, backend, and historical data should not be reflexively fixed. Valid outcomes include fix-now, fix-differently, skip, accept-risk, and record-as-lesson, each with a documented rationale.
- **Save the review report and read it before triaging large changes** — For bigger changes, persist the report and read it without time pressure to build a whole-picture understanding before entering finding-by-finding triage.
- **Review implementation against the plan with parallel adversarial agents** — One agent detects drift (MATCH/DRIFT/MISSING/EXTRA) and a second scans security, performance, and pattern compliance, feeding an interactive decision loop: fix, accept the risk, or save as a future rule.
- **Run controlled verification rituals inside execution skills** — Track touched files, run tests, pause at a manual verification gate, commit only named files (never `git add -A`), and record the commit SHA — a disciplined, auditable loop instead of unchecked autonomous edits.
- **Do solo review and a separate PR for every parallel change** — Parallel velocity is only safe when each change still passes review and opens its own PR; your review/decision bandwidth, not the agent count, sets real throughput.
- **Gather evidence from multiple sources before reading code** — No single signal gives a full diagnosis: monitoring locates the fault, logs show scale, and local reproduction separates frontend from backend, so the agent reads code with a target instead of blindly.
- **Verify the fix across every diagnostic layer that exposed the bug** — Verification should mirror investigation: if you diagnosed via four channels (monitoring, logs, repro, code), confirm the fix via all four — "works locally" is not enough.
- **Close planning with a verification gate of control questions** — A fixed checklist (access control, data model, one-sentence business rule, artifacts, scope feasibility, non-goals) catches missing decisions before they become foundations for everything built later.
- **Validate generated document quality before building on it** — Review a produced spec against concrete criteria (one concrete user, a feasible first flow, business logic as a concrete rule, explicit non-goals, a genuinely empty Open Questions) to catch a plausible-looking but hollow document.
- **Embed deterministic self-review gates before any disk write** — A machine-checkable review (frontmatter keys, section order, PRD coverage of every must-have requirement, acyclic dependency graph, table parity) acts as an embedded test that aborts the write on any failure with a specific error, protecting downstream tooling from silently propagated drift.
- **Make every rule checkable against a diff** — Re-read each sentence and ask "could a reviewer flag a diff against this?"; strike vague phrases like "best practices" or "be consistent" in favor of concrete patterns, thresholds, and named tools.
- **Name anti-patterns concretely with offered fixes** — Generic "your design has issues" feedback nullifies a quality gate because users dismiss it; name the specific defect (e.g. empty-CRUD, MVP-too-big) and offer concrete options.
- **Prefer soft gates that warn and record overrides over hard blocks** — A closing cross-check that informs, allows override, and records accepted gaps (status `warned`, mirrored into the next step's Open Questions) respects autonomy while preserving an audit trail.
- **Require every finding to trace to an inspected source** — Forbid invented facts: each rule or finding must cite a file:line, command, or commit you actually inspected, so the agent verifies instead of guessing and output stays trustworthy.
- **Enforce structural quality rules via self-review on generated artifacts** — Require each roadmap slice's Outcome to start with "user can…" and self-review that every must-have PRD requirement maps to at least one slice, so non-vertical slices and lost requirements are caught automatically.
- **Treat quality and overlooked details as the differentiator** — When anyone can generate complex tools, the edge comes from quality and the small UX/security details models almost always omit; focus human attention on what gets left out, not just what is produced.

## ❌ Anti-patterns

- **Trusting an agent's structural counts at face value** — Readers take "exactly four call-sites" or "only here" on faith, but the agent sees tokens, not named syntax nodes, so a missed file or uncounted method can silently break a risky change.
- **Merging AI code just because it runs and tests pass** — Working code can still solve the wrong problem, expand scope, bypass conventions, or ship tests that validate a faulty assumption; a green build is not merge-readiness, and approving without understanding the consequences lets risky changes through.
- **Accepting an AI-generated risk map without defending the decisions** — An elegant table can hide implausible scenarios, code-level descriptions instead of failure scenarios, or wrong impact/likelihood ratings; the human must correct it because the model lacks full business context.
- **Generating code (or knowledge) you never read** — Fully delegating decisions to the model ships fast but leaves you disoriented and unable to fix what breaks; for a knowledge base it is worse, because you lose the entire reason for building one.
- **Letting the agent act while you passively watch** — Without an approved plan, a multi-step operation that fails mid-way forces you to reconstruct intended steps from the agent's work history; Plan Mode documents intent and lets you catch steps the agent attempts that aren't in the plan.
