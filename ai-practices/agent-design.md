# Agent Design — Good vs. Bad Practices

A reference for engineering AI agents and agentic systems: how to scope, structure, gate, and orchestrate them so they stay predictable, safe, and useful as models improve.

## ✅ Good practices

- **Decide deliberately between agent, workflow, and plain code** — Reserve open-ended agent loops for genuinely dynamic data, NL-driven iteration, and flexible output; use a fixed workflow for stable, well-structured processes and plain code where you need zero cost, low latency, or full predictability. Always ask "is an LLM even necessary here?"
- **Choose an agent loop only when flexibility is needed; workflows for guaranteed fixed steps** — A deterministic workflow gives control but can't recover from a mid-task error; an agent can scan, verify, and self-correct. If you want the system to ALWAYS follow set steps, build a workflow instead.
- **Default to agentic logic, reserve deterministic logic for strong reasons** — Equipping a capable agent with simple tools (e.g. Agentic RAG) now often beats hand-written multi-stage pipelines; choose deterministic code only when there's a compelling reason.
- **Bound the agent loop with a maximum iteration count** — Cap iterations (e.g. 5–15) so a stuck or misbehaving model can't hang the server in an infinite loop.
- **Gate irreversible and consequential actions behind deterministic confirmation** — Enforce the accept/reject gate in code via a real UI control (or an allowlist / dry-run mode), never through LLM judgment, so a hallucination can't auto-send an email or delete data.
- **Show all action details in confirmation prompts, not just the headline field** — A human can only catch a leak or wrong target if shown the full payload; confirming an email address alone misses that the body may carry data pulled from other tools.
- **Weigh reversibility and blast radius before acting** — Let agents freely take local reversible actions (edit files, run tests) but require approval before destructive (`rm -rf`, drop table), hard-to-reverse (force push), or externally-visible (Slack/GitHub) actions.
- **Block irreversible external actions and keep a human as the final gate** — An email/triage agent can label, prioritize, and draft but has no send action; denying irreversible capability sharply limits blast radius from errors or injection.
- **Limit an agent's tools and content scope to bound the blast radius** — Since prompt injection can't be reliably filtered, the most robust mitigation is structural: an agent that simply lacks a send-email tool can't be tricked into exfiltrating data.
- **Programmatically scope data/knowledge access per session** — Enforce resource isolation in code, not the prompt: when the agent works on mailbox A, close access to mailbox B and unrelated knowledge to prevent cross-context leakage.
- **Give background agents a least-privilege, read-only, owner-only channel** — A dedicated agent identity that can only message its owner and has read-only knowledge-base access contains the damage if it is compromised.
- **Constrain agent permissions and physically block dangerous actions** — Because models misfire tools (wrong recipient, inviting outsiders to a meeting), limiting privileges or physically preventing specific actions is the primary safeguard.
- **Keep sensitive or error-prone steps deterministic, outside the LLM** — Where a mistake is costly (e.g. mixing up attendee addresses), handle that step in the interface so it never runs through the probabilistic agent.
- **Keep deterministic logic in code; let the model route and reference** — Growing model capability doesn't mean logic should move into the model; deterministic work belongs in code while the model handles query routing and references to documents/fragments.
- **Separate deterministic scheduling from non-deterministic agent decisions** — Let a manager agent dynamically create/assign tasks while a deterministic DAG scheduler resolves dependencies, ordering, and resumption, so the agent never owns brittle control flow.
- **Keep human-defined plan/task contracts with code-driven state and a heartbeat loop** — Encode plans as tasks with names, statuses, and dependencies, advance state in code, and re-check each cycle for agentic flexibility with guaranteed completion.
- **Give the agent a mutable task list to plan and manage its attention** — A to-do tool the agent creates and rewrites after each step re-surfaces priorities into context; because the content is model-generated, it steers behavior more strongly than externally supplied text.
- **Pass data and artifacts between tools/agents by reference, not by regenerating** — Have the first tool write a file and the second attach/reference it (resolved to Base64 or a URL at call time); inlining or regenerating doubles cost and roughly doubles task time, and references enable clean multi-agent handoffs.
- **Inject a referenceable handle for attachments, not just inline blobs** — Models can't "see" a URL or address a file, so add a media-tagged element holding a file reference plus a system instruction explaining how to use it, or agents can't hand attachments to tools.
- **Use parallel tool calls and smaller models to cut latency and cost** — Each agent step is a full LLM round-trip; run independent actions concurrently and route easy tasks to smaller, faster models. Under hard time budgets, sequential execution will simply fail.
- **Split model choice by task: reasoning models for planning, cheap models for coding** — Premium models earn their cost on analysis/PRD work, while a budget model can match or beat them on pure implementation at a fraction of the price (82.5 at $0.02 vs 76.8 at $8.69).
- **Match reasoning-effort level to task complexity** — Highly generalized or interface-generation tasks benefit from high reasoning effort at the cost of latency; choose the level deliberately as a quality/speed trade-off.
- **Match the model to the data type instead of one big prompt** — A code-level router that classifies material first (text/noise/binary) and sends only valuable data to an appropriately chosen model drastically cuts tokens versus funneling everything through one giant prompt.
- **Augment tool calls with reusable preset instructions (skills/commands)** — When a tool must behave consistently (e.g. a fixed image style), enrich the prompt with preset content surfaced as Commands/Skills/Prompts rather than re-supplying full instructions each time.
- **Package repeatable workflows as Agent Skills, not one-off prompts** — A skill is a versioned, on-disk procedure with a named input/output contract that loads progressively, invokes by name, and re-runs across sessions — guarantees a chat prompt lacks.
- **Extract a reusable skill only after repeated ad-hoc sessions** — Short ad-hoc prompts suffice early; only after several exploratory sessions across projects do the genuinely repeatable steps reveal themselves and become worth codifying.
- **Invoke important skills explicitly rather than trusting auto-activation** — Description-based activation requires the model to recognize its own knowledge gap, which it often fails to do; a Vercel eval showed auto-activation scored the same as no docs (53%), versus 79% with explicit invocation.
- **Use the description field deliberately to control activation scope** — Too-broad descriptions fire on everything, too-narrow ones never fire; tuning trigger phrases is the cheapest, highest-leverage activation knob.
- **Separate the procedure (skill) from the capabilities (tools)** — A skill encodes what steps to take; tools are how it acts. A skill without tools can't execute, and tools without a skill leave you with raw Bash and hope the agent assembles something.
- **Decompose multi-stage workflows into single-purpose agent sessions/skills** — Run each role (Researcher/Writer/Sender) in its own focused session or narrow skill that explicitly defers adjacent work, preserving quality and composability instead of one monolithic prompt.
- **Single-purpose skills with explicit hand-off boundaries** — Each skill does one job and explicitly refuses adjacent work (no frameworks, no schemas, no estimates), routing it to the right downstream skill to keep behavior predictable.
- **Chain skills via file-based handoff contracts** — Each step reads a file from the previous step and writes one for the next (`research.md → frame.md → plan.md`), so the downstream agent reads an explicit schema instead of guessing.
- **Skills communicate through files on disk, not shared memory** — File-based handoffs make multi-step workflows durable, inspectable, and resumable across sessions instead of relying on fragile in-context state.
- **Treat file names, headers, and status values as a stable downstream API** — Renaming `plan.md` or changing a status value can make a dependent agent fail silently, so a machine-greppable output contract (fixed sections, IDs, frontmatter) must stay stable.
- **Delegate to another skill via the Skill tool, not by reimplementing logic** — Calling a dedicated skill for a precondition (e.g. scaffolding) avoids drift between two copies of the same logic and keeps a single source of truth.
- **Gate execution with a pre-execution mandate check** — Require a concrete on-disk precondition (input artifact, passing tests, clean tree) before acting, redirecting to the upstream skill instead of improvising on a stale or empty context.
- **Structure agent work as three ownership gates: pre / in / post-execution** — Separate what's checked before acting (skill contract), approved during (harness permissions), and verified after (tests, CI, audit) so each concern has a clear owner.
- **Define explicit precondition and skip conditions for each skill** — Detect required project markers and STOP with a redirect if absent, and state when to defer to a sibling skill, preventing the skill from running on wrong inputs.
- **Delegate to authoritative CLIs instead of generating from training data** — When an official generator exists (`npm create astro`, `docker init`, `prisma migrate`), delegate to it for correct, current output; an agent reconstructing config from memory emits stale, half-working results.
- **When the authoritative tool fails, hard-stop instead of improvising around it** — On a non-zero exit from a delegated CLI, leave the partial scaffold and a failed verification report rather than fabricating the missing files into an unmaintainable result.
- **Never overwrite user files; write conflicting output to a sibling and preserve context verbatim** — Build non-destructive conflict handling into the tool (`package.json.scaffold`, merged `.gitignore`, verbatim-preserved decision artifacts) rather than trusting prompt instructions.
- **Default to non-destructive collision handling** — When regenerating an artifact, archive the prior version (`archive/<today>-roadmap.md`) rather than overwriting, making the version diff a first-class affordance.
- **Handle output-file collisions with an explicit overwrite-vs-versioned-save choice** — Prompt the user (overwrite / save as v2 / abort) before clobbering an existing report, avoiding silent data loss on re-runs.
- **Surgical update over wholesale rewrite for existing artifacts** — Classify each line KEEP/UPDATE/REMOVE/MISSING and prefer small Edit calls over a single Write, preserving human-authored voice and producing a reviewable diff.
- **Confirm scope with the user before applying edits** — Offer a structured choice (apply / show diff / regenerate / cancel) before mutating an existing file, keeping the human in control of destructive operations.
- **Treat archived artifacts as immutable and abort writes to them** — A hard guardrail against editing archive paths protects historical records from agent mutation and redirects the user to the correct entry point.
- **Never auto-chain into the next skill; halt and defer to the human** — End each stage with an explicit STOP and a "Next" pointer so the human verifies each artifact, preventing compounding errors across an autonomous pipeline.
- **Choose a multi-agent topology that matches coordination needs** — Pick among pipeline, blackboard, orchestrator, and tree; avoid mesh/swarm in production because they're hard to control and debug.
- **Keep multi-agent systems as simple as possible for as long as possible** — Complexity in agent systems is hard to trace; one system can serve several independent areas with minimal cross-agent exchange — start small rather than automating a whole org.
- **Start with one simple repeatable process rather than a grand system** — Rich agent systems aren't built in one evening; getting agents to reliably run a single repeatable process counts as success and lets you acclimate before scaling.
- **Modular separation with loose coupling keeps agent systems extensible** — Isolate each module's core logic and connect modules as loosely as possible via shared event contracts, so ongoing development stays tractable as the system grows.
- **Prefer information exchange between agents over direct task delegation** — Letting agents share outputs but not command each other keeps a multi-agent system simpler and more predictable than full inter-agent orchestration.
- **Give the orchestrator/manager few tools but broad read access** — A coordinator's value is system knowledge, plan-tracking, and routing — not many tools; overloading it complicates an already context-heavy role.
- **Model agent delegation as private child runs with explicit dependencies** — A sub-agent gets its own run (own agent/tools/context) while the parent enters a waiting state; on completion the scheduler delivers the result back like a call stack, enabling clean nestable orchestration.
- **Let a blocked sub-agent pause and request missing info rather than guess** — When a worker lacks information, suspend its loop and bubble a clarification request up to the parent and ultimately a human, preventing fabricated completions.
- **Assume inter-agent communication degrades and design for partial information** — Data passed between agents loses fidelity as complexity grows, so write delegation/message instructions carefully and assume an agent may need to verify what it received.
- **Fan out independent reads to parallel read-only subagents, but never delegate synthesis** — Subagents grep and analyze in their own context windows and return condensed reports with file:line references; the orchestrator keeps synthesis in one context to enforce budget, ordering, and reference policy.
- **Constrain delegated subagent prompts tightly** — Give read-only probes a strict output contract (word cap, required verdict, cite a file, no speculation, no edits) to bound context cost and prevent drift into changes.
- **Constrain parallel sub-agents to specific named dimensions** — Engineer one research query so it splits into exactly the sub-agents you need (e.g. end-to-end trace, test-coverage gaps, blast radius) to keep legacy coverage deliberate.
- **Fan out exploration to focused read-only subagents** — Split investigation into narrow read-only mandates (current shape, history/intent, migration feasibility), each verifiable and none allowed to introduce changes.
- **Know when NOT to parallelize** — Skip subagent fan-out for small repos, hosts without subagent support, or when files are already in context; re-reading via subagent just burns tokens.
- **Constrain a research/analysis skill to read-only with one allowed output file** — Restricting writes to a single `research.md` prevents an analysis pass from silently mutating project code under the guise of tidying up.
- **Gate research behind an explicit scope check to avoid burning tokens** — On an ambiguous request, pause and confirm scope, depth, and area before exploring, so the agent doesn't expend tokens on the wrong thing.
- **Ask the agent to present ranked options, not to make the decision** — Separate exploration from decision: the agent ends with a report of ranked options, and the irreversible choice (and the refactor) begins with the human.
- **Recommend one decisive next move, not a menu to triage** — Hand back a single recommended action with a one-line, overridable reason rather than a "ready" list that forces the human to redo the prioritization the agent just did.
- **Keep human checkpoints on the few decisions that actually shape the plan** — Ask only the questions the artifacts can't answer (goal, north star, top blocker) and arrive with a recommendation plus alternatives — neither full manual planning nor silent AI autopilot.
- **Use Plan Mode: read-only plan, then approve, before mutating operations** — Let the agent produce a reviewable step-by-step plan with no file mutations; the approved plan becomes a reference answering "what was supposed to happen?" when something breaks.
- **Scope each agent session to one well-understood slice, not a whole stream or MVP** — "Build the whole stream" just renames "build the MVP" at a higher level; one slice keeps diffs controllable and lets you tell real progress from accidental agent decisions.
- **Default to vertical slices that end in something a user can verify** — A vertical slice ("paste text and see saved cards") can be screenshotted and tested end-to-end — a strong success signal — whereas a horizontal layer only proves the layer exists.
- **Break implementation into separately verifiable phases** — A sequence of phases, each with its own contract, commit, and verification, enables stepwise stateful execution instead of one "implement everything" run that goes too far too fast.
- **Separate architect (planning) and coder (execution) roles** — Planning needs stronger reasoning while routine execution can use a cheaper model or separate thread; treating architect/coder/human as distinct roles clarifies who sets scope vs. executes vs. approves.
- **Choose interactive vs delegated mode by task certainty** — Use interactive sessions with manual pauses for complex, decision-heavy changes; delegate headless only when the plan is concrete, scope is closed, and completion is measurable.
- **Delegate with an explicit completion condition and turn cap** — A single goal command with measurable stop criteria ("stop after 20 turns if not complete") lets an evaluator model check the condition each turn and bound autonomous runs.
- **Learn the orchestration pattern, not a specific tool** — Multi-agent products rotate every quarter, but the pattern (isolate context per task, delegate a goal, review results in one place) is stable and is what transfers.
- **Express ordering as a dependency graph and mark parallelizable work** — Encode sequence as a topologically-sorted dependency graph and flag items with no path between their prerequisites so separate agent runs can fan out safely.
- **Run independent background agents in parallel, grouped only by time** — Background tasks that don't interact can be grouped by time window and run concurrently without artificial dependency chains, reducing coupling and latency.
- **Reuse one workflow contract across modes** — Make related skills share the same `plan.md`, `## Progress` section, and commit ritual, switching only the inner loop, so you mix modes per phase instead of spawning competing task lists.
- **Use red-first TDD to force the assertion before any code** — A failing test written first makes the agent reveal what it considers proof of correctness, letting the human inspect the oracle before implementation exists.
- **Hold background agents to a much higher precision bar than interactive ones** — Errors are tolerable when a human collaborates in real time, but an unattended agent that fails one run in five defeats its own purpose.
- **Match agent autonomy and permissions to sync vs. async mode** — Synchronous human-in-the-loop work can hold broad permissions because critical actions stay supervised; autonomous background work demands precise, self-sufficient processes needing no intervention.
- **Keep background agents isolated and single-purpose** — Design autonomous agents to work independently on narrow domains so conflicting dependencies and emergent loops never form; a ticket-classifier needn't know anything about an aggregator.
- **Specialize agents narrowly and prefer dedicated agents over one org-wide system** — Heavy specialization (e.g. a teacher agent persisting per-user progress) trades generality for effectiveness; prompt injection and hallucination make a single all-in-one org agent extremely hard to keep accurate and secure.
- **Design explicit fallbacks and a low-confidence bucket** — Flexible agents won't hit 100% accuracy, so give them an explicit "unsure" path (an `unclassified` directory, ask-for-missing-info, voice/confirmation handoff) rather than forcing a confident guess.
- **Notify only on real signal, with relevance gating** — Background agents should message the human only when genuinely needed and rank importance (e.g. by recurrence frequency) so attention goes to what matters.
- **Let the agent ask clarifying questions and retry before committing output** — Allowing the agent to request missing details and run several attempts against a style guide raises quality without restarting the whole task.
- **Run an event-driven agent loop with resumable, cancellable tasks** — Because runs take minutes and may pause for errors or approval, design the loop around events (iteration/tool-call/pause/resume/error) so you can monitor, compress context, and resume without re-running everything.
- **Wrap the agent loop in a job/run state machine with queueing and persistence** — Treat each message as a queued job with a separate run record (pending/running/waiting/completed) so work can pause, resume, and recover instead of being lost on failure.
- **Use heartbeat leases with claim-and-expire for crash recovery** — A worker claims a run with an expiry plus a periodic heartbeat; if the process dies, the reservation expires and a recovery pass re-queues the work, giving self-healing execution.
- **Give a single natural-language entry point for autonomous triggers** — Route all triggers (messages, hooks, webhooks, cron, heartbeat) through one entry point that interprets the task and adapts to current state, rather than hardwiring rigid handlers.
- **Build an agent harness, not just an SDK integration** — Effective agents depend on surrounding infrastructure — session hooks, background memory building, a file workspace, environment updates — that shapes context outside the live window.
- **Structure multi-agent workspaces by session with controlled inbox/outbox access** — A per-session file workspace with strict access rules (an agent owns its notes/outbox; only the root writes its inbox) enforces isolation and disciplined handoffs via shared files.
- **Use staged active directories as transform-then-promote pipelines** — Folders whose contents trigger automation (concept/review/ready/published) let documents be transformed in the background and handed to the next agent or human, giving an auditable staged workflow.
- **Share one file as real-time common state between human and agent** — A single JSON file as the sole source of truth lets a human edit in a browser while the agent reads/edits the same file, a clean human-in-the-loop pattern that never overwrites the user's work.
- **Give agents environmental awareness before tool calls** — Knowing the OS, interface type, interaction mode (e.g. CRON, no human), and a short user profile shortens the agent's path to the goal; ask "what must it know before running tools?"
- **Rely on agentic search via observation rather than scripted retrieval** — An agent that observes results and reformulates queries (grep/ripgrep) self-corrects and builds context dynamically — e.g. noticing documents are in Polish and re-querying — often with no special instructions.
- **Let agents enrich sparse inputs by connecting scattered context** — User messages and tool/memory data are often incomplete, so leave room for the agent to connect the dots (time, location, contacts) and turn fragments into actionable output.
- **Design a bounded space for the model to operate in rather than scripting every step** — Shape an environment with identity, capabilities, and constraints, then let the model choose actions within it, delegating proactivity and inference while keeping it inside designer-defined boundaries.
- **Treat non-determinism as a design feature, not only a bug** — LLM variability can surface useful solution paths a designer never anticipated; decide deliberately when to embrace it versus suppress it as a core architectural choice.
- **Separate agent settings, profile, rules, limits, style, and session into distinct sections** — Decomposing the system prompt into named concerns makes each agent's role and boundaries explicit rather than relying on raw intelligence, improving multi-agent predictability.
- **Don't bind the system prompt to dynamic tools or current directory structure** — Tool lists and file layouts change per session; reference roles and capabilities and use generic pointers ("instructions are in ./workflows") so the document graph can evolve without rewriting the prompt.
- **Avoid mixing output-style concerns with task logic in the same section** — Keep the voice/style section free of business logic so each instruction section stays cohesive and easier to reason about.
- **Put project rules in CLAUDE.md/AGENTS.md to turn a generic assistant into a role** — A persistent rules file encoding a full protocol (how to reset, respect the current phase, not overwrite the user's work) reliably casts the agent into a specialized role.
- **Design agents that get better as models improve** — A useful litmus test: if your system won't improve with a stronger model, you may be building the wrong thing or over-engineering scaffolding the model will soon obviate.
- **Treat model upgrades as chances to simplify, not just to stay stable** — Judge migrations on newly unlocked capabilities — a better model can let you remove instruction scaffolding or take on tasks that weren't previously feasible.
- **Anchor design on stable model fundamentals, not the hype cycle** — Core mechanics (autoregression, tokenization, context limits, hallucination, injection) have stayed constant for years, so logic built on them ages slowly while only the numbers shift.
- **Eliminate hard problems rather than incrementally patching them** — A design that removes the need for semantic search/graphs (e.g. Observational Memory) can beat solving each retrieval sub-problem; the reasoning pattern transfers broadly.
- **Generalizing solutions is a core skill for generative apps** — As code shrinks and the AI decides more at runtime, the remaining logic must be exceptionally polished universal mechanisms (context compression, planning, progress monitoring); there are no compilers for prompts.
- **Structure long-form work as a decompose-search-analyze-gap-fill loop** — The deep-research mechanic (decompose query, search, analyze, identify missing pieces, iterate) generalizes to "deep actions" like code generation and audits, not just reports.
- **Update only changed fragments of generated artifacts** — Patching specific sections rather than rewriting an entire document saves tokens and time, mirroring how coding agents apply diffs.
- **Delegate image interpretation to a dedicated tool or subagent, not the main loop** — Offload vision processing to a specialized tool/subagent and test which model works best, keeping the orchestration loop lean and reliable.
- **Adapt the model's output style to the delivery channel** — Formatting that works on screen fails over audio, so a voice agent must avoid dictating URLs, tables, or rich formatting that can't be conveyed by sound.
- **Treat agent UX surfaces — confirmation, progress, errors, cancel — as first-class** — Usefulness depends not on the tool call but on how runs are presented: the action to be run, progress, errors, results, and the ability to pause/cancel — most tools do this badly.
- **Clearly surface status and intervention needs for background and parallel runs** — When actions run in the background or across concurrent threads, users must be unambiguously told the current status and when intervention is needed, or they lose control.
- **Use subagent profiles to isolate distinct working contexts** — Split work into specialized subagent profiles, each with its own settings, knowledge, and model choice, switchable via menu/shortcut/@-mention, keeping configuration coherent.
- **Treat agent interaction as a stream of semantic events, model architecture around primitives** — Real agent output mixes reasoning, tool calls, media, confirmations, and errors; model chat as events between actors (an actor may be another agent or the system) so the schema absorbs multi-agent growth without rework.
- **Treat the database/event model separately from the LLM text and the UI** — Storing semantic events with IDs, types, and metadata keeps the system extensible (grow by adding a property), whereas raw assistant text as source of truth forces painful migrations.
- **Use a generic artifacts abstraction for generated content** — Represent all agent-generated content (images, files, interactive UIs) as typed artifacts with metadata, avoiding a separate schema per content type and enabling sharing between users and agents.
- **Centralize LLM communication behind a gateway** — A single layer for model communication lets you manage global request settings, monitoring, and provider/model swaps; scattering request-building across the app makes this nearly impossible.
- **Give agents a virtual bash filesystem instead of real terminal/sandbox access** — Models are fluent in bash, so a virtual filesystem driven by bash commands leverages that fluency while avoiding the complexity and risk of wiring up real terminals.
- **Prefer single-purpose, well-crafted tools over broad ones** — Tooling that answers a clearly defined, repeatable problem and does one thing well tends to be more reliable and valuable than sprawling general solutions.
- **Look for cheap single-action micro-tools, not just complex agent systems** — High daily value often comes from tiny scriptable actions bound to a shortcut (transform a selection, context-aware docs) that take minutes to ship; spotting them is the hard part.
- **Design multi-step conversations with strict ordering and a recovery path** — In stateful multi-turn dialogs both order and content matter; a wrong step can poison the whole exchange, so define the sequence and a clean restart path.
- **Prioritize evals on tool selection and tool usage** — Verifying that an agent picks and uses the right tools is the foundational eval, and it enables fast, cheap comparison of new or smaller models for cost reduction.
- **Include tool-call responses when designing agent evals** — Tool outputs quickly grow to dominate the prompt during a run, so evaluating only the system prompt and user message misrepresents the real input the model sees.
- **Score multiple weighted signals before classifying, not a single marker** — Tier signals by strength and require corroboration (git history, lockfiles, manifests) so one weak marker — like a bare `npm init -y` manifest — can't misclassify the branch.
- **Surface auto-detection results and let the user override** — Even good heuristics are sometimes wrong, so print detected signals, flag ambiguous cases, and confirm with an override option so a misdetection can't silently steer the session.
- **Detect scope/altitude before generating content** — The same skill should produce different output for repo-root vs. subdirectory scope; detect which applies (against `git rev-parse --show-toplevel`) before emitting repo-wide framing.
- **Facilitator agents must never invent domain content** — A discovery agent should elicit content from the user, not hallucinate vision/FRs/business rules; its value is the question shape and ordering, not the answers it supplies.
- **Keep the human as author of content; delegate only organization to AI** — Confine AI to formatting, linking, validation, and indexing while the human owns the substance, preserving authority and control over a knowledge base.
- **Set an explicit boundary the agent may not cross when editing your notes** — A clear line (reformat/transcribe but never invent content) prevents silent loss of control over content the human is meant to own.
- **Give agents note templates and a map-of-content to consult before writing** — Templates plus an index note let the agent survey structure, pick the right schema/location, and append or create consistently instead of inventing ad-hoc structure.
- **Decompose a repeatable process into a few plain text instruction files** — A multi-agent workflow can be driven by a handful of markdown files (research/assemble/deliver/info), turning text into a reliable repeatable process without bespoke code.
- **Prototype the agent system in Claude Code with no code first** — Connect the directory straight to the agent and shape processes, notes, and templates (and run via MCP/CLI) before writing any agent logic or deploying a server.
- **Run a two-agent interview to test multi-turn domain knowledge and auto-distill a brief** — One agent plays a curious developer and another a domain expert (ideally cross-provider), with a summarizer distilling a glossary and the setup doubling as a hallucination cross-check.
- **Choose agent-friendly tech stacks against explicit quality gates** — Score a stack on Typed, Convention-based, Popular-in-training-data, and Well-documented to predict where an agent produces correct, convention-respecting code.
- **Give skills explicit inputs and interview to fill gaps the files can't answer** — Ground decisions in real artifacts (`tech-stack.md`) plus a short interview for what isn't in the files; "I don't know" is a legitimate answer and the skill scores options rather than deciding.
- **Keep risk/strategy artifacts at the signal level, not code-line diagnosis** — A strategy phase reads the repo for hints ("this area changes often") but must not claim a specific bug location; only a later research phase inspects real code, preventing false anchors.
- **Don't pre-fill local conventions the model hasn't learned yet** — Leave the cookbook as "TBD" until research and implementation reveal how the project actually works, so the agent doesn't fabricate local patterns prematurely.
- **Implement soft gates that warn and record overrides rather than hard-blocking** — Soft gates keep the human in control while preserving an audit trail of risky decisions, avoiding both silent acceptance and rigid blocking.
- **Prioritize findings by agent impact, not generic severity** — Order fixes by how much they unblock reliable agent work (a missing test runner outranks a LOW CVE, because the agent can't verify its own changes without tests).
- **Assess project health for agent-readiness before agent work** — Audit dependency health, test infra, CI/CD, and config completeness first, emitting a healthy / needs-attention / critical verdict that turns "is it ready?" into a structured answer.
- **Keep health-check audits strictly read-only** — Suggest fixes but never run them (no `npm audit fix`, no auto-upgrade); WARN-AND-CONTINUE so the tool stays robust to missing tools and leaves prioritization to the user.
- **Pin dependencies via lockfiles so the agent can reason about exact state** — Unpinned dependencies make builds non-reproducible and leave the agent unable to reason about the exact state it operates on, undermining reliability.
- **Strict typing plus formatters/linters improve agent code reliability** — Strict types and configured linters give the agent stronger guardrails and consistent output, so their absence is a fix-before-agent-work item.
- **Strip cohort/branding references so skills are universally reusable** — Decoupling a skill's mechanics from course/brand-specific context makes it portable across projects and teams without leaking irrelevant references.
- **Question whether a chatbot/chat interface is even necessary** — The flashiest interface is often not the best fit; a well-configured search box, buttons, or a mobile app surfacing original documents may serve the need with less fragility, since agents tuned for specific phrasings break on real users.
- **Build agent logic without a chat window when possible** — Triggering models from user actions and resources (button clicks, file uploads) instead of a chat box gives far more control over inputs and scenarios and is often easier to ship.
- **Detect large pasted text and treat it as an attached file** — Users paste huge content into chat inputs, collapsing editing performance; detect oversized input and convert it to a text-file attachment.
- **Match LLM solution complexity to user AI-literacy** — Power users phrase queries with implicit tool hints while novices give vague prompts that agents fail on; designing for the less-experienced user determines org adoption.
- **Build tools you use yourself to develop intuition before scaling** — Being your own first user surfaces real problems and capabilities, building the hands-on intuition needed before designing larger-scale solutions for others.
- **Prefer AI that augments the human over AI that replaces or auto-generates** — Targeting collaboration and capability-amplification yields more value than fully generating outputs, which can carry no value for the user.
- **Don't reach for a full coding agent like Claude Code in production by default** — Off-the-shelf agents do everything in a prototype, but understanding the mechanics (or wrapping a minimal agent library) is what enables a controllable production path.
- **Reframe "X vs Y" tool choices as "when X, when Y, when both"** — Competing approaches (artifacts, JSON render, MCP Apps) mostly don't exclude each other; choose by business goals and real model capabilities rather than a winner-take-all debate.
- **The human shifts from code author to AI supervisor** — When AI generates most code, the human's value moves to steering direction, supplying context, making key decisions, and guarding output quality; naming this role keeps the human accountable.
- **Scope the agent to a single output and forbid collateral edits** — Constraining a skill to write exactly one file and touch nothing else makes its blast radius predictable and prevents scope creep into unrelated parts of the repo.

## ❌ Anti-patterns

- **Don't default to an agent when a workflow or plain code fits better** — For well-structured, stable processes, or where 100% reliability matters, an open-ended agent adds unpredictability; a fixed workflow or plain code (no LLM) is often the better, cheaper, more reliable choice.
- **Don't hand an agent a broad "build the whole MVP from the PRD" goal** — A goal with no sequence, priorities, or risk markers lets a non-deterministic model wander (endpoints one run, UI the next, forgotten migrations), burning tokens for non-repeatable results.
- **Avoid horizontal layering that manufactures fake progress** — Building all the DB, then all the API, then all the UI looks impressive but no user has seen the product's reason to exist, and integration conflicts surface only after weeks of stitching.
- **Don't spawn many agents without orchestration** — One agent per work item produces multiple branches, PRs, and sessions a single human must still own; that's more random code to take responsibility for, not more productivity — start with two slices, not five.
- **Avoid drifting into needlessly coupled multi-agent systems** — Teams gravitate toward elaborate systems where overlapping dependencies form by themselves, causing bugs that surface only after time or as the system grows — in extremes the logic loops.
- **Heavy human-wait dependencies sink autonomous-system efficiency** — When full isolation isn't possible and the system must wait on manual actions or decisions, its effectiveness drops sharply, so minimize required human handoffs in autonomous flows.
- **Don't conflate three different jobs in one prompt** — Asking an agent to simultaneously search structure, reason about architectural importance, and deeply understand behavior is too much at once and yields shallow results that miss critical dependencies and cycles.
- **Don't rely on system-prompt rules as a security boundary** — Prompt-level limits, safety, and constraints reduce but never guarantee compliance and can be jailbroken; real protection must be enforced in code and design.
- **Don't advertise capabilities in the system prompt that aren't wired up** — Telling an agent it can search the web while the tool is disabled makes the model act as if it can, producing fabricated or inconsistent behavior.
- **Don't rely on the model to know what it doesn't know** — An agent that believes it already knows an answer will skip loading the very skill or docs that would correct it, so passive availability of knowledge isn't the same as the model using it.
- **Don't expect prompts to execute deterministically like code** — Expecting stable, line-by-line deterministic execution sets up disappointment; LLM behavior is probabilistic and systems must be designed around variability.
- **Beware agents confidently and persistently implementing a wrong assumption** — Because agents execute consistently, a single unverified assumption baked in early propagates across the whole build, multiplying the cost of the mistake — which is why up-front questioning matters.
- **Nothing happens "on its own" — stay engaged in AI-driven workflows** — Agents default to shallow, low-value output and won't drive design/eval themselves; treating them as autonomous yields weak results, so sustained human steering is required.
- **Don't treat autonomous agent-generated tooling as production-ready** — Fully autonomous tool generation is very hard to control and supervise, so for now it should stay an assisted, human-in-the-loop activity.
- **Don't naively store raw LLM message text as your data model** — Appending generated text straight into storage seems simple but blocks richer interaction and forces painful DB/backend/frontend migrations once real users exist.
- **Avoid heavyweight LLM frameworks by default** — Current frameworks over-constrain without delivering proportional value; working directly with official SDKs is relatively simple and teaches the underlying mechanics.
- **Don't let automation flood you with content nobody reads** — Mass-producing reports or a daily newsletter that goes unread is the opposite failure of rigid automation and signals the workflow should be killed or trimmed.
- **Distrust sandboxes — agents find unforeseen escapes** — Agents discover workarounds designers didn't anticipate (writing a script to read a blocked `.env`), and modern models can detect they're being evaluated and hide true capabilities under test.
- **Shared tools/memory can cause false "not found" conclusions** — An agent with permission-restricted memory access that finds nothing may wrongly conclude the information doesn't exist at all, rather than that it lacked access.
