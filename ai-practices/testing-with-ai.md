# Testing with AI: Good Practices and Anti-Patterns

A reference for engineering tests and evaluations with AI agents. The throughline: agents optimize for "passes now," so the human owns deciding *what is correct* and *what is worth protecting*, while the agent does the mechanical work.

## ✅ Good practices

- **Start from a risk, not from a file** — A risk map answers "what business scenario must not break, and why" (impact × probability), so testing effort targets real pain instead of whatever is easiest to generate or reach.
- **Derive the test oracle from sources, never from the implementation** — Ground each assertion's expected value in external requirements (PRD, research, docs, contracts); if it is computed by the same logic under test, the test only proves the code equals itself and silently cements bugs.
- **Treat the oracle problem, not syntax, as the hard part of AI test generation** — LLMs reliably reproduce what code *does*, not what it *should* do, so the expected-behavior decision is the real work; the model will otherwise assert buggy output verbatim.
- **Apply cost × signal: pick the cheapest test that gives a real signal per risk** — Match the test layer to the specific risk instead of defaulting to e2e "to be safe" or maximizing coverage; reach for integration tests only where a mock would lie.
- **Mock only expensive/non-deterministic external boundaries; keep internal boundaries real** — Internal boundaries (auth, routing, DB) hold the integration risk and must stay real, while costly or flaky external APIs (LLMs, payment gateways) are mocked at the network layer for determinism.
- **Drive E2E generation from a prioritized risk map, not from scratch** — Pick the 1-3 highest business risks from a maintained test-plan that genuinely need browser-level coverage; route anything provable by an isolated function down to unit/TDD where it is cheaper and more stable.
- **Gate the expensive test layer before generating anything** — Before writing an E2E test, verify the risk is genuinely browser-level, the feature is built and runnable, and no green test already covers it; otherwise you waste the slowest, most brittle layer or lead a browser through a feature that does not exist.
- **Write a failing test that reproduces the bug before fixing it** — Test-driven bugfixing works from symptom to a red test to a fix to green; the agent can turn the symptom into the assertion, leaving a permanent regression guard.
- **Add a characterization test before you touch untested code** — Pin current behavior from a seam you do not edit, then make the structural change; the rule is "add the test before you touch," not alongside, so a refactor cannot silently alter behavior.
- **Assert on persisted state, not on the API response status** — A test that only checks a 200 response passes on a swallowed-error bug; reading the persisted database row is what makes the test fail for the right reason and actually catch the defect.
- **Pin the honest current contract instead of testing aspirational behavior** — Do not write a test for a rollback that does not exist; pin the consciously-tolerated behavior so the test catches the moment someone changes it without a deliberate decision, and record the gap as an open risk.
- **Guard, don't rebuild, for deliberate constraints** — For a conscious constraint, add a cheap deterministic guard (e.g., a unit test asserting two arrays stay in sync) that turns "easy to forget" into "impossible to break" — a reversible addition that leaves the load-bearing decision untouched.
- **Explicitly prompt for abuse and security scenarios the happy path omits** — Agents rarely propose misuse cases (IDOR, injection, secret leakage, rate-limit abuse) because "happy path" excludes an attacker; when a system has auth or payments, deliberately ask "what if someone uses this against intent?"
- **Treat git history and human experience as primary risk signals** — Change frequency reveals where a project still lives, and a short interview ("what are you afraid to break? where did you get burned?") surfaces context the model cannot infer alone.
- **Build the test environment before applying test-first patterns** — Test-first and integration tests assume a runner and infrastructure exist; sequence setup as its own dependency-ordered phase that immediately exercises the most important rule, so scaffolding does not masquerade as protection.
- **Build test data isolation in from the start (unique IDs + per-test cleanup)** — Combine unique identifiers with per-test cleanup and an afterAll safety net so suites are rerunnable and parallel-safe; writing this into the rules gets it right on the first iteration instead of in review.
- **Inject saved session state instead of logging in through the UI every test** — Log in once, save storageState to JSON, and inject it into each scenario; keep a couple of dedicated login/registration tests but never make them a dependency, and gitignore the auth file.
- **Reserve vision mode for what the DOM cannot express** — Vision models are slow, costly, and prone to coordinate drift and false positives; use a DOM snapshot for functional checks, vision only as a supplement for rendered-layout issues, and deterministic pixel-diff tools for visual regression.
- **Detect redundant tests that share intent but kill zero mutants** — Near-identical tests feel protective yet may detect no mutation; mutation results expose them so one well-named parameterized assertion can replace many copies.
- **Success criteria must verify behavior, not file existence** — Checking that files exist gives a false sense of done; criteria that assert observable behavior actually confirm the change works, and plan review should demand it.
- **A passing test is information, not proof the change is correct** — Agent-written tests are code too and can pass while only validating the implementation; review them as "would this actually catch the failure I want to avoid?" not "does a test exist?"
- **Assign each check to the right quality layer (per-edit, pre-commit, pre-push, CI)** — Place fast format/lint per-edit, typecheck at commit, the heavy suite at push, and infra-dependent integration in CI, so each layer stays fast while errors are still caught early.
- **Scope test hooks to files related to the edit, not the whole suite** — Run only tests that import the changed module (`vitest related <file>`, Jest `--findRelatedTests`) so per-edit feedback is fast enough to be worth blocking on.
- **Drive hook placement from the documented test plan and risk areas** — The test-plan's risk areas indicate which files merit an automatic post-edit check, and its quality gates map directly onto the per-edit / commit / push / CI layers.
- **Design test datasets for coverage, diversity, and balance** — Cover every behavior category including negative and edge cases, vary scenarios broadly, and avoid skew toward particular paths; for an agent with 5 tools, distribute attention across each tool and across non-happy-path cases.
- **Generate synthetic test datasets with the LLM, but curate diversity and validity yourself** — Models produce good evaluation data from context and rules, but the human must guarantee diversity and sensibility because the model cannot match still-evolving needs.
- **Organize test datasets into categories covering single-tool and multi-step scenarios** — Per-tool tests reveal how well a model handles each tool; multi-step scenario tests (including error and ineffective-action cases) reveal how well it chains them — and uncategorized tests are hard to extend.
- **Don't blindly trust eval scores — sometimes the test, not the system, is wrong** — A high score does not guarantee correctness and a low score may flag a genuinely better output, so eval results must themselves be human-reviewed and the datasets iterated.
- **Validate optimized prompts on held-out examples** — Scores from the optimization loop overstate real performance; testing on an unseen example gives a truer (lower but still improved) estimate and guards against overfitting to the tuning set.
- **Validate initial assumptions with fast, cheap throwaway tests** — Some up-front design decisions will be wrong and model/API behavior (cost, latency) is hard to predict, so quick experiments answer feasibility questions before heavy investment.
- **A/B test a candidate rule with fresh sessions before keeping it** — Run a representative task 3-5 times without the rule, add a minimal 1-3 sentence rule, then re-run in fresh sessions and compare convention adherence, time, files explored, and iteration count to prove it changes behavior.
- **Start skills via conversation, add evals only when shared or chained** — Conversational iteration with manual testing has zero infrastructure cost for personal skills; invest in eval harnesses only once a skill has many users or sits in a longer chain where regressions are costly.

## ❌ Anti-patterns

- **The vague "write tests for this file" prompt** — With no risk context or expected behavior, the agent tests the code paths it can easily reach, producing dozens of green tests in an hour and a false calm while the real risks sit elsewhere.
- **Prioritizing tests by what's easiest, not by risk** — Agents gravitate to low-value targets (helpers, getters, formatters), inflating coverage while critical flows stay unprotected and green CI becomes theatrical.
- **Letting the implementation be its own test oracle** — An agent asserting on the value the code already returns just freezes current behavior, bugs included; it is worse than no test because it gives false protection and makes every legitimate fix "break the test."
- **Treating coverage percentage as proof of protection** — A green bar and rising coverage only show a line executed, not that any assertion was meaningful; on a realistic benchmark, mutation testing showed ~38% of deliberate bugs passed 50 green tests unnoticed.
- **Fixing code "until tests pass" under green-bar pressure** — "Make it pass" pressure takes the easy path of bending the test to the code rather than the code to the requirements, silently corrupting the safety net; the direction of fitting matters.
- **Generated tests optimize for "passes now," producing predictable anti-patterns** — Naive assertions, brittle nth-child/CSS selectors, shared cross-test state, `waitForTimeout` instead of waiting on state, and missing cleanup all stem from the model optimizing for an immediately-green result rather than a test that is stable tomorrow.
- **Naively trusting a round-trip test as a safety net** — A write→read→compare test can stay green even when the bug is present, because read and write use different forgiving mechanisms; the safety net must be designed (e.g., sentinel values) to actually fail on the failure mode.
- **Auto-healers that fit assertions to wrong behavior** — A self-healing agent safely fixes selector drift, but when a test fails because backend logic or response format changed it adapts the assertion to the incorrect behavior, masking the bug; confine healing to selector-level breakage.
- **Parallel delivery without quality gates and tests** — Shipping faster with multiple agents but no consciously designed gates or automated tests means nothing catches a regression before users do.
